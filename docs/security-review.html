Security Review

A security audit of the Per-Ankh cloud-rewrite stack — the SvelteKit app under src/ and the Cloudflare Worker under cloud/. Each finding carries CVSS-style severity, file:line evidence, and a concrete remediation. Posture: principal-level Cloudflare review. Reviewed 2026-05-10.

Executive summary

Per-Ankh's security posture is, in aggregate, well above average for an app of this size. The hot paths — auth, authorization, SQL, blob storage, decompression — are uniformly careful: Discord OAuth with PKCE and timing-safe state comparison, opaque session tokens in KV with sensible cookie attributes, parameterized D1 queries throughout, server-side file hashing, gzip-bomb caps at every decompression site, deep PII strip for anonymous public reads, and a CSP that explicitly avoids script-src 'unsafe-inline' via SvelteKit's hash mode. No critical or high-severity exploitable issues were identified during this review.

The findings below are predominantly defense-in-depth gaps and one notable compliance-shaped issue: there is no account-deletion or data-export endpoint despite the worker collecting Discord ID and email at sign-in. The most consequential technical finding is a CORS allowlist that includes http://localhost:1420 in the production worker — currently inert thanks to SameSite=Lax, but it becomes immediately load-bearing the day cookies are loosened or a per-environment config split is missed.

Scope & methodology

In scope
Cloud Worker under cloud/ (~19 endpoints, D1/R2/KV); SvelteKit app under src/ (SSR, hooks, client API, parser); both projects' wrangler.toml, svelte.config.js, _headers, root tsconfig.json, prod package.json + lockfile audits.
Out of scope
Legacy web/ share viewer (frozen); bake pipeline (scripts/); historical git-log secret hunt; Cloudflare dashboard configuration (WAF rules, Bot Fight Mode, HSTS at edge, Web Analytics privacy); third-party services (Discord OAuth dashboard config, R2 bucket ACLs).
Method
Three parallel exploration agents mapped the attack surface (routing, auth, storage, CORS, rate-limiting, headers, dependencies). The reviewer then read every cited file end-to-end in the main context, drafted candidate findings with file:line evidence, and ran a per-claim verification pass against the actual code before any finding entered the report. Dynamic checks: npm audit on both projects; fast-xml-builder import-graph confirmation.
Severity
CVSS-style ladder (Critical / High / Medium / Low / Info). Hypothetical or theoretical concerns are filed as Info unless a concrete exploit path is described.

Findings

F-01 Medium No account-deletion or data-export endpoint
Component
Cloud Worker / Frontend account page
Evidence
cloud/src/auth.ts:419-441 (user upsert with email storage) · src/routes/account/+page.svelte (read-only profile + logout) · no DELETE /v1/users/me or GET /v1/users/me/export route registered in cloud/src/index.ts:548-680

Description

The Worker collects and persists discord_id, email, email_verified, display_name, and avatar_hash in the users table on first login, and accumulates upload metadata (game blobs in R2, player rosters in player_summaries, audit rows in events) on every subsequent action. The account page (/account) lets a user sign out but provides no mechanism to delete the account or export its data. No admin runbook step (under scripts/admin/) exists for self-service deletion either.

Impact

This is principally a compliance and trust issue rather than a vulnerability. GDPR Art. 17 (right to erasure) and CCPA give in-scope users a statutory right to deletion that the app cannot currently honor without operator intervention. Even setting aside the legal frame, long-lived PII without a delete path is bad data hygiene: a future breach of the D1 snapshot would expose every Discord ID + email that has ever signed in, including testers who only used the beta once.

Recommendation

Add a DELETE /v1/users/me endpoint that, in a single D1 transaction:

  1. Deletes all R2 objects under games/{user's game_ids}.json.gz and saves/{...}.zip.
  2. Deletes the users row (cascading deletes already handle games, player_summaries, collections, user_online_ids via FK constraints).
  3. Tombstones the discord_id in events.metadata (overwrite to "[deleted]") so the audit trail keeps row count integrity without retaining PII.
  4. Deletes any active sessions for the user from KV (requires either a per-user session index or a re-check of discord_username in session.ts:88-99 — the existing allowlist re-check at auth.ts:548-552 already gives you the second option for free).

For data export, a GET /v1/users/me/export that streams a ZIP of the user's blobs + a JSON summary is the standard shape. Add the link to the account page UI.

F-02 Low Production CORS allowlist includes http://localhost:1420 with credentials
Component
Cloud Worker CORS
Evidence
cloud/wrangler.toml:82: ALLOWED_ORIGINS = "https://per-ankh.app,http://localhost:1420" · cloud/src/util.ts:62-82 echoes the Origin and emits Access-Control-Allow-Credentials: true when matched

Description

The production worker's CORS allowlist explicitly trusts http://localhost:1420 in addition to the production frontend origin. Any page running on that origin can send credentialed cross-origin requests to api.per-ankh.app and read responses.

Impact

Today the issue is fully neutralized by SameSite=Lax on the session cookie (cloud/src/session.ts:127) — modern browsers will not attach the cookie to a JS fetch from a different eTLD+1, regardless of CORS posture. So an attacker page served from http://localhost:1420 on the victim's machine can make the request but receives unauthenticated 401s. The risk is two coupled scenarios where the mitigation erodes:

  1. A future change to SameSite=None (e.g. to support an embedded third-party widget) makes the issue immediately exploitable. There is no codified guardrail tying the CORS allowlist to the cookie attributes.
  2. A local attacker who controls a process bound to port 1420 on the victim's loopback (malware, a compromised dev tool, a co-tenant on shared infrastructure) is granted credentialed CORS access. This is outside the standard browser threat model, but the allowlist explicitly opts in to it.

Recommendation

Split CORS config per Cloudflare environment. Use [env.production] and [env.dev] sections in wrangler.toml so the production worker never sees localhost in its allowlist: 

[env.production.vars]
ALLOWED_ORIGINS = "https://per-ankh.app"

[env.dev.vars]
ALLOWED_ORIGINS = "https://per-ankh.app,http://localhost:1420"

Deploy production with wrangler deploy --env production. Run local dev against --env dev. As a belt-and-suspenders, assert at handler startup that no allowlist entry uses an http:// scheme when running under the production binding (D1 ID is a convenient discriminator).

F-03 Low Anonymous read rate-limit bypass via spoofed scraper User-Agent
Component
Cloud Worker — GET /v1/games/:id anonymous path
Evidence
cloud/src/games.ts:71-86 defines the scraper allowlist; games.ts:1215-1247 applies it before the per-IP limit and audit log

Description

Anonymous reads of public games are rate-limited at 200/hour per IP via D1 audit events, but the limit is skipped entirely for requests whose User-Agent begins with one of eight scraper prefixes (Discordbot/, Slackbot/, Twitterbot/, facebookexternalhit/, etc.). The header is trivially attacker-controlled; any client can send User-Agent: Discordbot/1.0 and bypass the limit.

The in-code comment (games.ts:69-70) acknowledges this explicitly: "Trivially spoofable, but the data is public — this is availability defense, not security."

Impact

Cost amplification more than data exposure. The PII-stripped blob is already public, so an attacker scraping public games at high rate gains no additional information. They do force Worker decompression + re-serialization on every cache miss (cache TTL is 60s edge, games.ts:1291), incurring Worker CPU + R2 reads. With ~1000 public games rotated across, an attacker could trigger ~60k origin hits/hour — within Workers paid-tier limits but visible on the bill.

The exemption also defeats the audit trail: anon_read events are not logged for UA-matched requests (games.ts:1218 gate), so post-incident analysis of an abusive crawl is harder.

Recommendation

Two layered improvements:

  1. Keep the audit log unconditionally. Move the INSERT INTO events call out of the isScraperUA branch so every anonymous read is recorded, even if it's exempt from the rate-limit decision. The events table is internal D1 (already on the PII deny-list for Logpush), so the volume cost is bounded.
  2. Verify scraper identity via reverse DNS or signed fetch. Discord's bot publishes its outbound IP ranges and the Reporting API; Cloudflare's edge can apply a "Known Bots" managed rule that runs reverse-DNS verification on User-Agent: Discordbot before it reaches the Worker. Move the exemption out of the app and into a Cloudflare WAF rule that you can be confident about. Same for Slackbot (publishes its IP range publicly) and Twitterbot.
F-04 Low State-changing endpoints don't validate request Content-Type
Component
Cloud Worker — all POST/PATCH/DELETE handlers under /v1/auth/*, /v1/games/*, /v1/collections, /v1/csp-report
Evidence
cloud/src/auth.ts:174,250,566 · cloud/src/games.ts:694,1311,1437 · cloud/src/collections.ts:69 · cloud/src/csp.ts:88 — all call request.json() / request.formData() / request.text() without first asserting request.headers.get("Content-Type")

Description

The CSRF model the codebase relies on (correctly documented at games.ts:1298-1303) is: SameSite=Lax blocks cross-site cookie attachment on POST forms, and any cross-origin JS fetch with a non-simple Content-Type triggers a CORS preflight that the allowlist rejects. The model holds, but it has one quiet seam: a cross-site <form enctype="text/plain" method="POST"> can submit JSON-shaped bodies as a CORS-simple request, skipping preflight entirely.

Today that seam is closed because SameSite=Lax blocks the cookie on cross-site POSTs (Chrome's "Lax+POST" 2-minute exemption is being phased out, but historically existed). Once SameSite is no longer load-bearing for some other reason — e.g. embedded use cases, mobile webview semantics, third-party iframes the team explores later — strict Content-Type validation is the only remaining wall against a hand-crafted cross-site form submission.

Impact

Defense-in-depth only at present. Not currently exploitable.

Recommendation

Add a one-liner gate near the top of each state-changing handler: 

const ct = request.headers.get("Content-Type") ?? "";
if (!ct.startsWith("application/json")) {
  return errorResponse("Content-Type required", 415, cors, "UNSUPPORTED_MEDIA_TYPE");
}

For POST /v1/games require multipart/form-data; for POST /v1/auth/logout keep it permissive (it's idempotent and body-less). The CSP report endpoint can stay as-is (it's intentionally promiscuous about input shape) but should reject empty/unknown content types with 415 rather than logging the odd payload — see also F-05.

F-05 Low /v1/csp-report is unauthenticated and unrated
Component
Cloud Worker — cloud/src/csp.ts
Evidence
csp.ts:88-141 accepts any POST with a 64 KB body cap, parses JSON, emits one structured log line per violation. No auth, no per-IP counter, no body-shape allowlist.

Description

CSP violation reports are processed without authentication or rate limiting. The endpoint is intentionally public — browsers POST here directly when the page CSP triggers, with no cookies and no preflight — but every request that survives the 64 KB body cap turns into one or more Logpush event lines via logEvent("warn", "csp_violation", ...).

Impact

Logpush cost amplification. An attacker can sustain ~1 Worker request/sec from a single IP (Cloudflare's free-tier ceiling) with a 64 KB body containing N synthetic violations. Each violation yields one log line. The endpoint is global, so traffic from many IPs fans out without a circuit breaker.

Reasonable real-world cost: with the Reporting API format, one POST can contain an array of report objects; normalizeReportingApi emits one log per entry. A 64 KB body of ~200-byte report objects yields ~300 log lines per request. Sustained ~300 lines/sec is meaningful at Logpush ingest rates.

Recommendation

  1. Add a per-IP rate limiter (Cache API counter, same pattern as checkDownloadRateLimit in index.ts:198-214). Reasonable budget: 60 reports/IP/hour. CSP reports are one-per-violation, not high-frequency under normal use.
  2. Cap the per-request violation count after normalization (e.g. take the first 10 entries of any Reporting API array). The first few are diagnostically useful; everything beyond that is fan-out.
  3. Reject bodies with an unknown Content-Type rather than logging csp_violation_unknown_format (csp.ts:131). Browsers always send one of the two known content types; logging unknowns is asking for unbounded noise.
F-06 Low CSP-report URL fields not PII-scrubbed before Logpush
Component
Cloud Worker logging
Evidence
cloud/src/csp.ts:138 emits the whole NormalizedViolation object as log fields. log.ts:32-42's PII_KEYS deny-list contains online_id, discord_id, username, email, access_token, code_verifier, session_token, app_key, body — but not document_uri, blocked_uri, or source_file.

Description

The CSP violation logger surfaces document_uri, blocked_uri, and source_file as top-level log fields. These URLs are partly under the page author's control (own-origin documents) but blocked_uri in particular reflects whatever URL the browser tried to load — which on a compromised page can be attacker-controlled and can contain query parameters with PII (?email=..., ?session_id=...) the attacker crafts specifically to land in your logs.

Impact

Low-grade Logpush pollution / potential third-party PII at rest. Not a vulnerability in Per-Ankh per se; more a "we are the unwitting downstream of someone else's bad day."

Recommendation

Strip URL query strings and fragments before logging. The path + origin are sufficient for diagnosing CSP issues: 

function stripQuery(u: string | null): string | null {
  if (!u) return null;
  try {
    const url = new URL(u);
    return `${url.origin}${url.pathname}`;
  } catch {
    return u.split("?")[0].split("#")[0];
  }
}

Apply to document_uri, blocked_uri, source_file, and the new documentURL/blockedURL Reporting API equivalents.

F-07 Low SSR /share/* redirect does not validate id format
Component
SvelteKit SSR hook
Evidence
src/hooks.server.ts:66-72 — Response.redirect(`https://legacy.per-ankh.app/share/${id}`, 302) where id = event.url.pathname.slice("/share/".length)

Description

Any request matching /share/* on the main frontend is 302-redirected to legacy.per-ankh.app with the remainder of the path appended verbatim. The id is not validated against the 21-character nanoid shape, so requests like /share/%2e%2e/admin redirect to https://legacy.per-ankh.app/share/%2e%2e/admin. When the browser follows the redirect it decodes %2e%2e and resolves the path to https://legacy.per-ankh.app/admin.

Impact

Minimal — the legacy share viewer at legacy.per-ankh.app is a frozen static SvelteKit app that has only /share/[id] routes; arbitrary paths return a standard SvelteKit 404. There is no admin surface to land on. The concern is mainly that the main app is propagating attacker-controlled path bytes into a redirect to a different origin — a habit worth not having, since the legacy host may grow new routes later or be repurposed.

Recommendation

if (event.url.pathname.startsWith("/share/")) {
  const id = event.url.pathname.slice("/share/".length);
  if (!/^[A-Za-z0-9_-]{21}$/.test(id)) {
    throw error(404, "Not found");
  }
  return Response.redirect(`https://legacy.per-ankh.app/share/${id}`, 302);
}
F-08 Low Transitive fast-xml-builder advisory (not invoked, but pinned)
Component
Frontend dependency tree
Evidence
npm audit --omit=dev on the root package reports fast-xml-builder <= 1.1.6 with GHSA-5wm8-gmm8-39j9 (high) and GHSA-45c6-75p6-83cc (moderate). Source: transitive of fast-xml-parser ^5.7.2. grep across src/ + cloud/ confirms zero imports of fast-xml-builder, XMLBuilder, or j2xParser — the app uses only the parser side.

Description

The fast-xml-parser package ships fast-xml-builder as a sibling package in the same monorepo. The app uses only XMLParser (see src/lib/parser/parse-xml.ts:40), so the vulnerable builder code is not on any code path. The audit warning is real but not exploitable in this app.

Impact

None today. The risk is that a future code change adds an XML-building path (e.g. to generate share OG previews as SVG, or to render an XML sitemap) and inherits the vulnerable version without anyone noticing the warning has become live.

Recommendation

Bump fast-xml-parser when its next minor release pulls in a patched fast-xml-builder. In the meantime, add a one-line lint or pre-commit grep that fails the build if any source file imports fast-xml-builder, so the advisory can't quietly become exploitable.

F-09 Low Transitive cookie low-severity advisory
Component
Frontend dependency tree (SvelteKit)
Evidence
npm audit on the root package reports cookie < 0.7.0 with GHSA-pxg6-pf52-xh8x (low) — "accepts cookie name, path, and domain with out of bounds characters." Pulled in transitively by @sveltejs/kit.

Description

The Node cookie package, used by SvelteKit for cookie serialization, does not validate the name / path / domain fields of cookies it serializes. An attacker who could influence any of those three fields could inject CR/LF bytes that split into a separate header. Per-Ankh's SvelteKit code never sets a cookie with an attacker-influenced name, path, or domain — all cookie sites use hardcoded names — so the issue is not exploitable here. The Worker (which sets the real session cookie) implements cookie serialization inline (cloud/src/session.ts:123-135), bypassing this package entirely.

Impact

None today. Same forward-looking concern as F-08: future code that takes a cookie name from user input would inherit the issue.

Recommendation

The advisory's fix is a major-version SvelteKit bump. If the team is already on a recent enough @sveltejs/kit minor that ships cookie ^0.7.0, an npm update may suffice. Otherwise, add the bump to the next dependency-sweep cycle.

F-10 Low Discord redirect_uri not server-side allowlisted
Component
Cloud Worker — OAuth start + callback
Evidence
cloud/src/auth.ts:189-197 accepts any string for redirect_uri; auth.ts:311 only checks that the callback's value matches what the start request sent. There is no comparison against a fixed list of acceptable values.

Description

The Worker trusts whatever redirect_uri the frontend sends and forwards it to Discord's /authorize endpoint. Discord's own dashboard allowlist is the only thing preventing an attacker who controls a frontend call from setting redirect_uri=https://attacker.example/. As long as the Discord dashboard is configured correctly (with only https://per-ankh.app/auth/callback registered) this works — but it relies on a single point of failure outside the codebase.

Impact

Defense-in-depth gap. Exploitable only if (a) the Discord dashboard allowlist is misconfigured to permit additional redirect URIs, or (b) an attacker independently controls a frontend already permitted by CORS (which would already be game-over).

Recommendation

Maintain a server-side allowlist parallel to the Discord dashboard's. The list is short (one prod URI, optionally one dev URI) and rarely changes: 

const ALLOWED_REDIRECT_URIS = new Set([
  "https://per-ankh.app/auth/callback",
  "http://localhost:1420/auth/callback",  // gate on env, see F-02
]);

if (!ALLOWED_REDIRECT_URIS.has(redirectUri)) {
  return errorResponse("redirect_uri not allowed", 400, cors, "REDIRECT_URI_DENIED");
}

Apply at both handleDiscordStart (auth.ts:189) and as a second check at handleDiscordCallback after the equality compare (auth.ts:311).

F-11 Low PII deny-list scrubs only top-level log keys
Component
Cloud Worker logging
Evidence
cloud/src/log.ts:113-128 — scrubPii walks Object.entries(fields) once; nested objects are passed through unchanged.

Description

The redaction layer correctly catches { discord_id: "abc" } at the top level, but a hypothetical handler that wrote { metadata: { discord_id: "abc" } } would surface that field verbatim into Logpush. Current handlers are disciplined — every PII field is either at the top level (caught) or in a JSON.stringify(...) blob going to D1 (intentional, never to logs) — but the invariant is not enforced by code.

Impact

Defense-in-depth. Nothing leaks today.

Recommendation

Make scrubPii recursive. A shallow walk of arbitrary objects is cheap (the fields object is bounded by handler discipline already) and removes the need to remember which call sites have a chance of containing nested PII: 

function scrubPiiDeep(value: unknown): { out: unknown; redacted: boolean } {
  if (Array.isArray(value)) { /* recurse, accumulate */ }
  if (value && typeof value === "object") {
    let redacted = false;
    const out: Record<string, unknown> = {};
    for (const [k, v] of Object.entries(value)) {
      if (PII_KEYS.has(k) && v != null) { out[k] = "[REDACTED]"; redacted = true; }
      else { const r = scrubPiiDeep(v); out[k] = r.out; redacted ||= r.redacted; }
    }
    return { out, redacted };
  }
  return { out: value, redacted: false };
}
F-12 Info CSP style-src 'unsafe-inline'
Component
SvelteKit CSP config
Evidence
svelte.config.js:50 — "style-src": ["self", "unsafe-inline"]; comment at line 7-10 acknowledges as a known compromise pending CSP-report data.

Inline styles are widespread in the codebase (e.g. src/lib/game-detail/GameDetailView.svelte:180,183,…: style="background-color: #2a2622;"), bits-ui injects them via portals, and ECharts emits them dynamically. Moving to hash mode for styles would be a multi-PR effort. The current posture is the standard pragmatic compromise for a Svelte 5 + ECharts app. script-src remains strict (hash mode with no unsafe-inline), which is the meaningful win.

Recommendation: Keep the current setting. Use the CSP-report endpoint (after F-05 is addressed) to enumerate the actual inline-style hashes that ship and revisit tightening on a one-time pass.

F-13 Info No HSTS / COOP / COEP set by the app
Component
SvelteKit hooks + repo _headers
Evidence
No Strict-Transport-Security, Cross-Origin-Opener-Policy, or Cross-Origin-Embedder-Policy set in src/hooks.server.ts:79-90 or _headers.

HSTS is typically delivered at the Cloudflare edge (dashboard → SSL/TLS → Edge Certificates → HSTS); the codebase delegates that responsibility. COOP/COEP are not required by current functionality (no SharedArrayBuffer use, no popups receiving auth callbacks).

Recommendation: Verify HSTS is enabled at the Cloudflare edge for per-ankh.app with at least a 6-month max-age and includeSubDomains. Consider adding Cross-Origin-Opener-Policy: same-origin in hooks.server.ts for free XS-leak isolation; the OAuth flow uses top-level redirects, not popups, so this is non-breaking.

F-14 Info Email stored from Discord but never read by the app
Component
Cloud Worker — user upsert
Evidence
cloud/src/auth.ts:50 requests identify email scope; auth.ts:417, 423-441 stores email and email_verified in the users table. No grep hit for any code that reads either column outside the upsert.

Email is collected at sign-in and persisted in D1 but never used: no notification pipeline, no account recovery flow, no admin lookup tooling reads it. This is unnecessary PII at rest and a wider email scope on the Discord OAuth consent screen than the app actually needs.

Recommendation: Drop the email scope from OAUTH_SCOPE (auth.ts:50) and remove the column writes. If you later add email-based notifications, opt-in re-collection is cleaner than holding long-lived emails for a feature that doesn't exist yet. Tightens the consent screen, which is a trust signal for new testers.

F-15 Info login_denied audit row stores Discord ID + username at rest
Component
Cloud Worker — allowlist gate
Evidence
cloud/src/auth.ts:392-403 writes { discord_id, discord_username } into events.metadata JSON for every rejected login.

The intent (moderation trail of who tried to sign in but wasn't on the allowlist) is sensible. The implementation persists Discord PII in the same D1 instance as the rest of the app, indefinitely (subject only to the 2%-per-request 90-day cleanup at index.ts:106-118). The in-line comment correctly notes these rows are not shipped to Logpush, but a future analytics export could sweep them up.

Recommendation: Either (a) hash the discord_id with a server-side pepper before storing — same identity comparison if the same user retries, no reversible PII at rest — or (b) shorten the retention for event_type = 'login_denied' to 7 days specifically (separate from the 90-day general cleanup).

F-16 Info Session cookie Domain=per-ankh.app implicitly trusts any future subdomain
Component
Cloud Worker session cookie
Evidence
cloud/src/session.ts:115-119,131-132 sets Domain=per-ankh.app on the session cookie in production. The comment explains it's necessary for SSR at per-ankh.app to read a cookie set by api.per-ankh.app.

The design is correct for the current two-host topology (apex SSR + api. Worker). The side-effect is that any future subdomain — legacy.per-ankh.app already exists for the frozen viewer; a future blog., docs., preview. would inherit the same scope — receives the session cookie on every request. A vulnerability on any such subdomain (XSS, host takeover) would expose the cookie. Even though it's HttpOnly, the subdomain could trivially make authenticated calls against api.per-ankh.app.

Recommendation: Track this as a known-and-accepted constraint. When introducing any new subdomain, either move it to a separate eTLD+1 (e.g. a different apex), or audit it as critical-impact alongside the main app. Document this in a SECURITY.md or the architecture doc.

F-17 Info Legacy handleDelete 404 vs 403 enumeration oracle
Component
Cloud Worker — DELETE /v1/share/:id
Evidence
cloud/src/index.ts:494-503 returns 404 when the share id is unknown and 403 when the id is known but credentials are wrong.

The status-code split lets an attacker who has guessed a share_id distinguish "doesn't exist" from "exists but I don't have the delete token." The share id is a nanoid(21) (~126 bits of entropy), so brute-force enumeration is infeasible; the oracle is observable only when the attacker has independently obtained a candidate share_id (e.g. a leaked share URL).

Recommendation: Collapse both branches to 404 with the same body. Same pattern is already followed correctly in handleGamePatch (games.ts:1328-1330) and handleGameDownload (games.ts:1524-1527). The legacy endpoint is being decommissioned, so this can be deferred until that work concludes — noting it here for completeness.

F-18 Info handleGameDetail "Blob missing" reveals D1/R2 inconsistency
Component
Cloud Worker — GET /v1/games/:id
Evidence
cloud/src/games.ts:1250-1253 · same shape at games.ts:1555-1556 for the download path.

When a game_id row exists in D1 but the R2 blob is missing (e.g. after a failed cleanup or a partial restore), the response body is {"error":"Blob missing","code":"BLOB_MISSING"}. This reveals to a non-owner public viewer that the game existed and was uploaded — which is information the access-control branches at lines 1202-1208 carefully avoid disclosing elsewhere. The blob-missing case bypasses that hedging because it returns 404 with a distinct code/message.

Recommendation: Return the same 404 body as the "row not found" branch ({"error":"Not found","code":"NOT_FOUND"}). Keep the structured log line (logError("blob_missing", ...) would be useful for operators); just don't surface the distinction over the wire.

F-19 Info tsconfig lacks noUncheckedIndexedAccess
Component
Both projects' TypeScript config
Evidence
tsconfig.json (root) and cloud/tsconfig.json — strict: true set; noUncheckedIndexedAccess not set.

Without this flag, arr[i] is typed T instead of T | undefined. The codebase has several places where indexed access is dereferenced without checking — e.g. auth.ts:160-164 generateState writes into a fixed-size buffer (safe); games.ts:436 uses roster.map(...) correctly. No exploitable issue was identified, but the compiler is currently silent about a class of off-by-one / empty-array bugs it could catch.

Recommendation: Turn the flag on in a focused PR; expect a small fix-up pass to add explicit ?? defaultValue or guards. The Worker is the higher-value target — the cloud Worker has more index-access surface in the multipart / multi-row insert paths.

F-20 Info /v1/stats endpoint not rate-limited
Component
Cloud Worker — cloud/src/stats.ts
Evidence
No countEventsSince / Cache-API counter checks in stats.ts:21-100.

The endpoint runs five D1 aggregations per request, all parallelized and all scoped to session.user_id. There is no per-user rate limit. The endpoint is authenticated, so an attacker would need a valid session; the only abuse path is a legitimate user spamming their own dashboard with refreshes, which costs D1 reads but no PII exposure.

Recommendation: Optional. If D1 read budget becomes a concern, cache the response per-user for ~60s at the worker layer (or add a per-user rate-limit of, say, 600/hour as a quiet ceiling). Lower priority than F-05 which is unauthenticated.

F-21 Info Generic session cookie name
Component
Cloud Worker session cookie
Evidence
cloud/src/session.ts:26 — const SESSION_COOKIE = "session";

Using the literal name session on a shared apex domain (Domain=per-ankh.app) increases the odds of a name collision if another service is ever deployed on a sibling subdomain that also chose session. Browsers concatenate same-named cookies from multiple sources; the server then has to guess which one is theirs. This is more a robustness consideration than a security one, but ties into F-16's subdomain trust footprint.

Recommendation: Rename to __Host-pa-session if you can drop the explicit Domain attribute (the __Host- prefix requires omitting Domain, requires Secure, requires Path=/, and ties the cookie to one specific origin — strong defense against subdomain mishaps). This conflicts with F-16's apex-cookie requirement, so the trade is: pick subdomain-isolation (__Host- on each host that needs a session, no shared cookie) versus SSR-cross-origin-sharing (current state, Domain=apex). The former is the stronger posture if your topology allows it; in the current setup, even renaming to pa-session reduces collision risk without changing semantics.

Strengths worth preserving

These are not findings, but they are non-obvious good decisions the team should resist regressing.

  • SvelteKit CSP in hash mode — no script-src 'unsafe-inline'. Avoids the single most common XSS-amplification path in SSR apps.
  • Deep PII strip on anonymous public reads (games.ts:157-190): the recursive walk catches future schema changes that could put online_id in new locations, not just where it currently lives.
  • Username allowlist re-checked on every /v1/auth/me (auth.ts:548-552): access revocation propagates within the session TTL window without requiring a session-invalidation pass.
  • nanoid(32) session token with 30-day KV TTL: opaque, high-entropy, server-side revocable.
  • PKCE + state validated with crypto.subtle.timingSafeEqual and KV entry deleted unconditionally on callback to make replay impossible (auth.ts:284-300).
  • Single chokepoint cookie forwarding for SSR with an explicit allowlist of response headers passed through (hooks.server.ts:17-21,49-59).
  • Structured logging with AsyncLocalStorage + a PII deny-list applied at emit time (log.ts) — request_id correlation works without threading a context argument through every signature.
  • Decompression with streaming size accumulator — gzip-bomb defense that stops decoding as soon as the cap is breached, rather than buffering and checking after (util.ts:147-179).
  • D1 batched upload writes with INSERT OR REPLACE + FK cascade for re-imports, with documented degraded-state semantics on failure (games.ts:990-1051).
  • Path-parameter regex validation at the dispatch layer (index.ts:548-680) — every :id is shape-checked before the handler runs.
