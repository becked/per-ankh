Tournament Branch Security Review

A security audit of the tournament branch — the multi-commit feature work that adds Swiss / championship-bracket tournaments to Per-Ankh. Scope covers everything new on the branch relative to main: 13 cloud-Worker source files under cloud/src/tournament/, 8 new D1 migrations (0006–0013), the tournament-link guard on the games delete path, the beta-allowlist gate threaded through auth, the public read endpoints, the admin handlers and admin CLI, and the SvelteKit tournament UI under src/lib/tournament/ + src/routes/tournaments/. Reviewed 2026-05-17.

Executive summary

No high-confidence security vulnerabilities were introduced by this branch. The tournament feature is well-defended. Authorization is centralized through a small set of wrappers (authedTournament, requireBetaOr404, authedBetaSession) that consistently apply the beta-allowlist gate before admin checks and return 404 TOURNAMENT_NOT_FOUND for non-beta users — hiding feature existence, not just access. The integration test suite includes a sweep (cloud/test/integration/tournament/beta-gate.test.ts) that asserts the 404 behavior across the main tournament endpoints.

Every dynamic-SQL surface I examined builds its UPDATE fragments from schema-stripped keys (Valibot's v.object drops unknown keys from the parsed output before reaching the SQL builder), with values universally parameter-bound via .bind(). Every cross-tournament resource access (matches, slots, rounds) is re-checked against the URL's tournament_id, defeating the "swap an ID in the body" IDOR pattern. RNG separation is correct: a deterministic mulberry32 drives pairing shuffles and map-assignment (where determinism is desirable for replayability), while IDs and tokens use nanoid / crypto.getRandomValues. PII (discord_id, email) is not returned by any tournament endpoint.

The branch also closes a previously-unguarded delete path: migration 0013 pairs a worker-side 409 LINKED_TO_ACTIVE_TOURNAMENT guard in handleGameDelete with a D1 BEFORE DELETE trigger that nulls tournament_matches.game_id. The two-layer design prevents the prior race where R2 blobs were dropped before the FK aborted the D1 DELETE.

Scope & methodology

In scope
All changes on tournament relative to main as of commit 13b3f08. Cloud Worker tournament module (cloud/src/tournament/: admin, authz, public, data, player, bracket, pairing, standings, limits, maps, canonical-map-options, canonical-maps, rng), migrations 0006–0013, the games.ts delete-guard, beta-allowlist wiring in auth.ts, validation schemas (cloud/src/schemas/tournament.ts), routing in cloud/src/index.ts, the admin CLI (scripts/admin/commands/tournament.ts), and the SvelteKit tournament UI.
Out of scope
Pre-existing surfaces not touched by this branch (re-reviewed in the 2026-05-10 cloud-rewrite review): OAuth flow internals, session-cookie attributes, base CSP / SSR security headers, parser worker, share-blob shape for non-tournament games. Cloudflare dashboard configuration (WAF, edge HSTS, Bot Fight Mode) and third-party services (Discord OAuth, R2 ACLs) are also out of scope.
Method
A sub-agent enumerated candidate findings file-by-file across the changed surface, tracing every state-changing route from dispatcher → wrapper → handler → DB. The reviewer then spot-checked the dynamic-SQL UPDATE builders, the absence of {@html} sinks in tournament Svelte components, and the schema-strip guarantee from Valibot in the working tree before signing off. Per the false-positive filtering rules in the review prompt, theoretical issues and defense-in-depth gaps not introduced by this branch were not reported.
Severity
CVSS-style ladder (Critical / High / Medium / Low / Info). Only findings at confidence ≥8/10 are reported. The Evaluated & cleared section documents what was looked at and why each item is not a vulnerability — useful as a record that the area was examined, and as a reference for future review of code added on top of this branch.

Findings

None. The tournament branch did not introduce any high-confidence vulnerability that meets the impact ≥ Low & confidence ≥ 8 bar applied by this review.

Evaluated & cleared

Each entry below is a candidate attack path the review considered and rejected, with the specific code that defeats it. These are not findings — they are an audit trail so that future review of code added on top of this branch can see what was checked and what the assumed defense is. If any of the cited defenses is later weakened, the corresponding attack becomes viable.

C-01 Cleared Dynamic UPDATE SQL fragments in admin handlers
Component
cloud/src/tournament/admin.ts — patch handlers
Evidence
admin.ts:687-715 (handlePatchTournament) · admin.ts:860-878 (handlePatchSlot) · admin.ts:1316-1329 (handleRetroEditMatch)

What the pattern looks like

for (const [key, value] of Object.entries(patch)) {
  if (value === undefined) continue;
  fragments.push(`${key} = ?`);
  binds.push(value);
}
await env.SHARE_DB
  .prepare(`UPDATE tournaments SET ${fragments.join(", ")} WHERE …`)
  .bind(...binds)
  .run();

Why it's safe

patch is the .output of a Valibot v.safeParse(schema, body) call (schemas in cloud/src/schemas/tournament.ts). Valibot's v.object strips unknown keys at parse time, so the key reaching the SQL string is always one of a fixed, code-defined column set. Values are always parameter-bound. An attacker cannot smuggle a SQL fragment via a key name or value.

What would make this unsafe later

Migrating to v.looseObject or switching to a manual Object.assign-style parser without an allowlist would break the guarantee. If the schema construction changes, re-verify.

The only string-concatenated piece is a fixed-shape predicate gated on a Valibot-validated filter value drawn from an explicit allowlist set (allowedStatus). Every value that ends up substituted into the query is parameter-bound. No injection surface.

C-03 Cleared Caller-controlled key lookup into CANONICAL_SCRIPT_OPTIONS
Component
cloud/src/tournament/admin.ts — map-options reconciliation
Evidence
admin.ts:96

Why it's safe

The script key reaching CANONICAL_SCRIPT_OPTIONS[script] is only used after being filtered through allowed.has(script), where allowed is a Set populated from input.allowed_map_scripts. That field is validated against StrictMapScriptSchema (an enum of canonical MAPCLASS_* identifiers). Strings like __proto__, constructor, etc. cannot survive that validator, so no prototype-pollution-style read or write is possible.

C-04 Cleared Beta-allowlist gate coverage across tournament surfaces
Component
Wrappers in cloud/src/tournament/authz.ts; dispatch in cloud/src/index.ts
Evidence
cloud/test/integration/tournament/beta-gate.test.ts:45-148 — sweep test

Why it's safe

Every tournament-namespaced route in the dispatcher resolves through one of three wrappers — authedTournament, requireBetaOr404, or authedBetaSession — each of which calls requireTournamentBeta before any handler logic runs. The beta-gate.test.ts sweep asserts that a non-beta authenticated user receives 404 with body TOURNAMENT_NOT_FOUND across the 17 endpoints it covers. The four endpoints it doesn't cover by name (reorder slots, patch slot, delete slot, the games-tournament-link branch) all go through the same wrappers in source. The 404 (rather than 403) hides feature existence rather than just access.

C-05 Cleared Cross-tournament IDOR via swapped match / slot IDs
Component
cloud/src/tournament/admin.ts + data.ts
Evidence
data.ts:187-197 (loadSlotInTournament) · admin.ts:1177,1231 (round-tournament guards) · public.ts:550 · cloud/test/integration/tournament/retro-edit.test.ts:205-243

Why it's safe

Every admin handler that accepts a slot_id or match_id in the URL or body re-validates that the target row's tournament_id matches the URL's :tournament_id path parameter. A user with admin rights on tournament A who supplies a body referencing a slot from tournament B receives a 404, not a successful write. The retro-edit integration test exercises this cross-tournament path explicitly.

C-06 Cleared Retro-edit can't crown an arbitrary slot
Component
cloud/src/tournament/admin.ts — handleRetroEditMatch
Evidence
admin.ts:1254-1282

Why it's safe

handleRetroEditMatch validates that the supplied winner_slot_id is one of the two slot IDs attached to the match being edited and that the slot belongs to the URL's tournament. An admin cannot retroactively crown a slot from a different match or a different tournament.

C-07 Cleared Setup-phase tournaments hidden from non-admins
Component
cloud/src/tournament/public.ts
Evidence
public.ts:148-163 (list LEFT JOIN gate) · public.ts:273-285 (setupGateHides)

Why it's safe

Every public read endpoint (detail, standings, bracket, rounds, matches, match detail) calls setupGateHides, which returns 404 if the tournament is in setup phase and the caller isn't an admin. The list endpoint enforces the same constraint via a LEFT JOIN predicate. Draft tournaments are not enumerable.

C-08 Cleared Games delete + tournament-link trigger interaction
Component
cloud/src/games.ts + cloud/migrations/0013_tournament_match_game_set_null.sql
Evidence
games.ts:1923-1939 (worker-side 409 guard) · migration 0013 (BEFORE DELETE trigger)

Why it's safe

The pre-fix bug (R2 blobs deleted before the D1 DELETE aborted on an FK violation) is closed by a two-layer fix: the worker-side guard returns 409 LINKED_TO_ACTIVE_TOURNAMENT if the game is linked to a match in a non-complete tournament, before any destructive R2 or D1 work runs. Only after a tournament reaches complete does the guard let the delete through, at which point the migration 0013 BEFORE DELETE trigger nulls tournament_matches.game_id, preserving the historical winner_slot_id and status. The trigger can't be exercised against an active tournament because the worker-side guard runs first. New test delete-tournament-linked.test.ts covers both branches.

C-09 Cleared PII held back from tournament responses
Component
cloud/src/tournament/public.ts, player.ts, admin.ts
Evidence
No discord_id or email column appears in any tournament response shape; standings / bracket payloads explicitly enumerate fields.

Why it's safe

Tournament endpoints return discord_username (the public Discord display handle, by design) and the internal user_id nanoid. They do not return discord_id, email, online_id, or other PII. online_id stripping in share-blob / game-detail paths is unchanged by this branch.

C-10 Cleared RNG separation
Component
cloud/src/tournament/rng.ts + ID/token sites
Evidence
rng.ts (mulberry32 + FNV-1a seed); nanoid import sites across cloud/src/

Why it's safe

The deterministic mulberry32 PRNG seeded by FNV-1a of a tournament / round string is used only for pairing shuffles and per-round map assignment — places where determinism is desirable (so a regenerated round is reproducible and an operator can replay decisions). The output of this RNG does not gate authentication, mint tokens, or generate IDs that need to resist guessing.

Security-sensitive randomness — match IDs, slot IDs, round IDs, tournament IDs, session tokens, OAuth state / PKCE verifier — uses nanoid (CSPRNG-backed) or crypto.getRandomValues directly.

C-11 Cleared CSRF on new state-changing routes
Component
cloud/src/tournament/admin.ts — JSON body parser
Evidence
admin.ts:285-329 (parseJsonBody Content-Type assertion) · cloud/src/session.ts (SameSite=Lax) · cloud/src/util.ts:62-82 (CORS allowlist)

Why it's safe

All POST/PATCH/DELETE tournament routes pass through parseJsonBody, which requires Content-Type: application/json — preventing the classic HTML-form CSRF gadget. The session cookie remains SameSite=Lax, so a cross-site fetch from a non-allowlisted origin will not attach the cookie. The CORS allowlist (production: https://per-ankh.app) is the same as for the rest of the app. There is a dedicated content-type.test.ts integration test asserting the JSON-only behavior.

C-12 Cleared XSS in tournament UI
Component
src/lib/tournament/**/*.svelte + src/routes/tournaments/**/*.svelte
Evidence
No {@html ...} or innerHTML assignment in any tournament Svelte component (verified via grep on the working tree).

Why it's safe

User-controlled tournament fields — name, description, slug, slot discord_username, map-script options — are rendered via Svelte's auto-escaping mustache ({...}) only. No component opts into the raw-HTML sink, and the parent CSP from svelte.config.js remains strict on script-src in any case.

C-13 Cleared Admin CLI uses dynamic SQL via sqlStr
Component
scripts/admin/commands/tournament.ts
Evidence
CLI builds SQL with string concatenation and shells out to wrangler d1 execute as the operator.

Why it's safe

The admin CLI runs locally under the operator's wrangler login session. It is not exposed over the API and its inputs come from CLI flags typed by the operator. This is not part of the runtime attack surface. (Style-wise it would still be nicer to parameterize, but it's not a vulnerability.)

Functional bugs noted in passing

These are not security findings — they don't enable any unauthorized access or data exposure. Listed here because the review surfaced them and they look worth fixing as ordinary follow-ups.

  • cloud/src/tournament/admin.ts:1188-1198 (handlePatchMatchMap): doesn't intersect the new map_script against the tournament's allowed_map_scripts at the handler layer, despite the schema comment at cloud/src/schemas/tournament.ts:26 implying it does. Stored string is auto-escaped on display, so there is no XSS path; this is a constraint-enforcement gap, not a vulnerability.
  • cloud/src/tournament/admin.ts:1444-1476 (handleTransitionChampionship): the override_ranks array isn't checked for duplicate slot references. An admin who specifies the same slot twice as an advancer creates duplicate championship slots all referencing the same source. Admin-only; functional bug.
  • cloud/src/tournament/admin.ts:1325-1329 (handleRetroEditMatch game_id patch): any nanoid21 that satisfies the FK to games is accepted. Pointing at someone else's private game doesn't bypass that game's ACL — game-detail handlers enforce their own auth — so there's no IDOR / data-exposure path. Admin power that could surprise; consider scoping to games the admin owns.

What was not reviewed

  • Pre-existing surfaces untouched by this branch — already covered by the 2026-05-10 cloud-rewrite review.
  • Cloudflare dashboard / edge configuration (WAF rules, edge HSTS, Bot Fight Mode, R2 bucket ACLs).
  • Third-party services and their dashboard configuration (Discord OAuth app, billing).
  • Dependency-supply-chain audit beyond what npm audit reports as part of preflight.
  • Performance / DoS / rate-limiting capacity — explicitly excluded by the review prompt.

Strengths worth preserving

  • Centralized auth-wrapper composition. authedTournament / requireBetaOr404 / authedBetaSession consistently apply the beta gate before admin checks. This is the kind of small-surface authorization plumbing that's easy to extend without breaking existing routes — keep new tournament endpoints going through one of these.
  • 404-not-403 for feature hiding. Non-beta users get TOURNAMENT_NOT_FOUND rather than FORBIDDEN, so the feature's existence isn't observable to outsiders during the private beta. The sweep test in beta-gate.test.ts documents the contract.
  • Schema-stripped keys in dynamic UPDATE builders. Building SQL fragments from Valibot's parsed output keys (rather than from raw request-body keys) is the right pattern. The shape is easy to copy-paste correctly the next time a patch endpoint is added — but it must be the parsed output, not the raw body.
  • Cross-tournament re-checks on every nested resource. Patterns like loadSlotInTournament and the round.tournament_id !== tournamentId guards close the standard IDOR pattern and are cheap to apply consistently.
  • Two-layer destructive-op guard. The handleGameDelete 409 + migration 0013 BEFORE DELETE trigger pattern is a good template: the worker-side check gives the user a clean error code, and the trigger keeps DB consistency at the constraint layer if a future code path is ever added that bypasses the handler-side check.
  • RNG separation. Deterministic mulberry32 for pairings / map assignment (replayable, auditable) and crypto-secure nanoid / getRandomValues for IDs and tokens. Keep this distinction explicit if anyone adds another randomized subsystem.
  • JSON-only state-changing routes. parseJsonBody's Content-Type assertion closes the simplest CSRF gadget without needing per-route token plumbing. The dedicated content-type.test.ts integration test pins this behavior so it doesn't regress.
