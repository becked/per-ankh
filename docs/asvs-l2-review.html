ASVS 5.0 Level 2 Coverage Review

A coverage-style security review of the Per-Ankh cloud-rewrite stack against the OWASP Application Security Verification Standard, version 5.0, Level 2 ("Standard" — applications handling sensitive data). This complements the per-PR exploit-focused review by measuring posture against a published bar rather than searching for concrete bugs. Reviewed 2026-05-17 against branch tournament.

Executive summary

Per-Ankh sits in good shape against ASVS 5.0 L2. The vast majority of applicable requirements are Met with concrete evidence — universal SQL parameterization, Valibot input validation, hash-mode CSP, server-side file hashing, gzip-bomb caps, PII-deny-list logging, single-IdP OAuth with PKCE and timing-safe state validation, multi-tier rate limits, and consistent cross-tournament IDOR defenses. No Gap represents a concrete exploitable vulnerability; the gaps are split roughly evenly between missing-documentation items (data classification, key management, upload policy, logging inventory, vuln remediation SLA) and defense-in-depth opportunities (session lifecycle, redirect-follow on external fetches, antivirus on user-uploaded ZIPs, cookie host prefix, admin session termination).

Of the 13 Gaps, the single most actionable code change is 15.3.2 — cloud/src/auth.ts:345,374 calls Discord with the default redirect: "follow"; setting it to "error" is a one-line hardening against a misconfigured 3xx redirecting OAuth secrets to an attacker host. The highest-leverage operational gap is 7.4.5 — there is no admin tooling to revoke a user's KV sessions; the existing ./per-ankh admin security script (scripts/admin/commands/security.ts:241) explicitly acknowledges this as a known limitation.

Scope & methodology

Standard
OWASP ASVS 5.0.0 (released 2025). Requirement text fetched from github.com/OWASP/ASVS/tree/master/5.0/en. Level 2 = all L1 requirements + all L2 requirements per row.
In scope
Cloud Worker (cloud/src/** + cloud/migrations/** + cloud/wrangler.toml); SvelteKit app (src/**, svelte.config.js, src/hooks.server.ts, repo-root _headers); deploy CLI (scripts/prod/checks/**); admin CLI (scripts/admin/**).
Out of scope
Legacy share viewer (web/, frozen); bake pipeline (scripts/bake-*); parser worker (src/lib/parser/**, deferred for its own pass against V2 + V5); Cloudflare dashboard config (TLS ciphers, HSTS, WAF, Bot Fight); third-party services (Discord OAuth app, billing). V9 Self-Contained Tokens (no JWTs in use), V17 WebRTC (not used).
Method
Five parallel sub-agents covered ASVS chapters in balanced buckets (Identity, Access & API, Input & Frontend, Crypto/Comms/Config, Data & Operations). Each fetched the official ASVS 5.0 markdown for its chapters, filtered to L2 requirements, and produced a per-requirement coverage table with file:line evidence. The reviewer then verified 100% of Gap rulings against the working tree and spot-checked Met rulings, reclassifying where verification disagreed (notably V7.4.4 — logout visibility — was reclassified Gap → Met after confirming CloudHeader ships on every authed route).
Severity
Four-state coverage: Met / Partial / Gap / N/A. A Gap is a missing defense; a Partial is a defense that exists but is incomplete (asymmetric application, undocumented, narrowly scoped). No CVSS-style severity here — that's the per-PR review's job.

Gap list

All Gap items, prioritized by risk × leverage. Each links to its per-chapter row for the full evidence trail. Code / configuration gaps

15.3.2 Outbound fetch() to Discord uses default redirect: "follow"
cloud/src/auth.ts:345-349 (token exchange) and cloud/src/auth.ts:374-376 (user fetch) call Discord without setting the redirect option. A misconfigured 3xx (DNS hijack, mTLS strip, future Discord-side accident) could silently redirect the request — carrying the client_secret body or the user's access_token bearer header — to an unintended host.
Fix: add redirect: "error" (or "manual") to both fetch() options. One-line change per call site.
5.4.3 No antivirus / structural scan on user-uploaded ZIP saves before re-serving
User-uploaded .zip save files are stored in R2 and served back via GET /v1/games/:id/download (cloud/src/games.ts:1995-2010). The download path is auth-gated to the owner or to public-flagged games, but no AV scan runs at upload time. ASVS L2 expects untrusted files served to clients to pass a scanning step.
Fix: integrate a Workers-compatible scan (e.g. via Cloudflare Workers Browser Rendering / a managed AV microservice), or document the accepted risk that downloads are auth-gated and saves are treated as opaque user-content (the parser runs client-side; the server never executes the bytes).
7.4.5 No admin tooling to terminate a user's KV sessions
./per-ankh admin nuke-user deletes the D1 row but does not enumerate KV session entries; the existing security script explicitly acknowledges this: scripts/admin/commands/security.ts:241 — "KV sessions are NOT cleared — stale tokens will 401 on next call". The lazy 401 happens only on endpoints that re-read the user row; endpoints that trust session.data.user_id directly accept orphan sessions until their 30-day TTL.
Fix: add a D1-side session index (or scan the KV with a prefix list if the namespace stays small enough) so nuke-user can call SESSIONS_KV.delete for every active token. Same mechanic enables a self-service "log out everywhere" surface later (V7.5.2).
7.1.2 Concurrent-session policy is undefined and unenforced
createSession (cloud/src/session.ts:33-47) writes a new KV entry per login; existing sessions for the same user are not enumerated or invalidated. Multiple devices/browsers each get an independent token, all valid until their own TTL.
Fix: either document a "no per-user session cap" policy explicitly, or implement a cap. Pairs naturally with 7.4.5 (same D1 index unlocks both).
7.5.2 Users cannot list or terminate their own sessions
/account only exposes logout for the current session. No GET /v1/auth/sessions or DELETE /v1/auth/sessions/:id endpoint exists. A user who suspects an old browser session was compromised has no self-service way to revoke it.
Fix: after 7.4.5 lands, surface the session list on /account with per-row revoke buttons.
6.8.4 / 10.3.4 Authentication strength / recency claims from Discord are not validated
Per-Ankh uses Discord OAuth 2.0 (non-OIDC), which returns no acr / amr / auth_time claims. The app has no signal whether the user actually completed Discord 2FA, and inherits whatever recency Discord considers acceptable.
Fix: either migrate to Discord's OIDC endpoint and request id_token (Discord supports OIDC discovery), or document the accepted risk and the inherited-MFA stance in docs/security/auth-model.md.
3.3.3 Session cookie does not use the __Host- prefix
cloud/src/session.ts:135 sets Domain=per-ankh.app to share the session cookie between per-ankh.app (SSR) and api.per-ankh.app (Worker). The __Host- prefix is incompatible with the Domain attribute, so it cannot be used given the current two-host topology. The same risk is already covered by F-16 in the prior review.
Fix: document the cross-subdomain trade-off as an accepted constraint in docs/security/auth-model.md alongside guidance that any new subdomain inherits session-cookie exposure (and therefore needs to be treated as a critical-impact surface).

Documentation gaps

14.1.2 No data-classification & protection-level matrix
The PII_KEYS deny-list in cloud/src/log.ts:32-42 is a de-facto inventory but lacks a formal taxonomy mapping each data type (discord_id, email, online_id, IP, save bytes, game metadata) to classification and protection requirements.
Fix: add docs/security/data-classification.md with a per-type matrix (sensitivity, encryption, retention, logging stance, access controls). Carries 14.1.1 and 14.2.4 (Partials) along with it.
11.1.1 / 11.1.2 No cryptographic key-management policy or inventory
Per-Ankh has very few keys (DISCORD_CLIENT_SECRET, INVITE_CODE, Cloudflare-managed at-rest keys for D1/R2/KV) but the policy governing rotation, owner, and lifecycle is implicit. The crypto inventory (SHA-256 + PKCE S256, CSPRNG via crypto.getRandomValues, nanoid, timingSafeEqual) is also undocumented.
Fix: add docs/security/key-management.md combining the policy (rotation cadence, owner, vault = wrangler secret put) and inventory in one page.
15.1.1 No documented vulnerability-remediation SLA for third-party dependencies
./per-ankh prod preflight runs npm audit --audit-level=high and blocks deploy on findings (scripts/prod/checks/npm.ts:54-104), but the target SLA against which the gate operates is unstated.
Fix: add a paragraph to docs/cloud-deploy-plan.md defining remediation timeframes (e.g. critical within 7 days, high within 30).
5.1.1 Upload file policy is undocumented
Size caps (MAX_BLOB_COMPRESSED=10MB, MAX_BLOB_DECOMPRESSED=50MB, MAX_ZIP_BYTES=50MB) and the no-server-parsing posture are implemented (cloud/src/games.ts:54-56) but not gathered into a single spec section.
Fix: add an "Upload file policy" subsection to docs/cloud-rewrite-spec.md listing accepted parts, MIME assumptions, size caps, hashing, and the operator's malicious-file response posture.

Partials list (high-leverage)

Selected Partials worth fixing alongside the Gap list. Full per-chapter Partial details live in the chapter tables.

3.4.4 API Worker responses omit X-Content-Type-Options: nosniff
SSR responses set nosniff in src/hooks.server.ts:76; the Worker's jsonResponse helper (cloud/src/util.ts:84-95) does not. Trivial fix.
3.5.1 / 3.5.2 Content-Type CSRF check is asymmetric across endpoints
parseJsonBody in cloud/src/tournament/admin.ts:285-329 rejects bodies without Content-Type: application/json (CSRF defense-in-depth). cloud/src/auth.ts uses a separate readJsonBody helper without the check (auth.ts:159-161), and handleGamePatch / handleCollectionCreate / multipart upload likewise don't enforce it. SameSite=Lax is the actual load-bearing CSRF defense today; the Content-Type assertion is the belt to that suspenders, and it's currently only on some pants. Fix: promote parseJsonBody to a shared helper and route all JSON endpoints through it.
7.2.4 New session on login doesn't invalidate prior sessions
createSession (cloud/src/session.ts:38) mints a fresh token; the prior cookie value is overwritten client-side but the prior KV entry continues to live until its independent 30-day TTL. A leaked token from an earlier login remains valid. Fix: after 7.4.5 lands, delete prior sessions on login.
14.3.2 Anti-cache headers are uneven across authed JSON responses
Owner game-detail sets Cache-Control: private, no-store (cloud/src/games.ts:1724-1725). /v1/auth/me, /v1/games list, and tournament endpoints set no Cache-Control at all. Modern browsers don't typically cache JSON without explicit headers, but defense-in-depth wants the explicit assertion. Fix: set Cache-Control: private, no-store in jsonResponse (or a sibling helper for authed JSON).

Strengths worth preserving

Patterns this review independently validated that are worth keeping stable as the app grows.

  • Universal SQL parameterization. Every D1 call uses .prepare(sql).bind(...). Dynamic UPDATE builders iterate Valibot-stripped keys, never raw request-body keys. (V1.2.4, V15.3.3)
  • Auth-wrapper composition. authedTournament / requireBetaOr404 / authedBetaSession consistently apply beta-allowlist before admin checks; 404-not-403 hides feature existence. (V8.2.1, V8.4.1)
  • Cross-tenant re-checks on every nested resource. URL tournament_id is authoritative; nested slot_id / match_id / round_id in the body are re-bound to that tournament before mutation. (V8.2.2)
  • Two-layer destructive-op guard on handleGameDelete: worker-side 409 + migration 0013 BEFORE DELETE trigger. The trigger can't fire on active tournaments because the worker-side check runs first. (V2.3.4)
  • Hash-mode CSP (no script-src 'unsafe-inline'), explicit frame-ancestors 'none' + X-Frame-Options: DENY on SSR, strict connect-src allowlist. (V3.4.3)
  • Multi-layer rate-limiting: per-user, per-IP, and global counters on every mutating surface, with a UPLOADS_ENABLED kill switch. (V2.4.1, V15.2.2)
  • RNG separation. Deterministic mulberry32 for pairing/map assignment (replayable), nanoid + crypto.getRandomValues for security IDs/tokens. (V11.5.1)
  • PII deny-list at log emission. Even if a handler accidentally attaches PII to a field, scrubPii catches it before the line ships, and tags the line with pii_redaction: true. (V16.2.5, V16.4.1)
  • OAuth flow rigor. PKCE S256, timing-safe state comparison, single-use oauth_pending, confidential client auth, minimum scope, open-redirect defense duplicated on server + client. (V10.1.x, V10.2.1)
  • Streaming gzip-bomb defense with size accumulator that cancels the stream on overflow, rather than buffering and checking after. (V5.2.3, V15.2.2)
  • AsyncLocalStorage request_id correlation surfaced to clients via X-Request-Id for bug-report attribution without threading context through every signature. (V16.2.4)
  • Top-level catch returns only {error, request_id} — no stack traces, class names, or error messages from the runtime leak to clients. (V16.5.1)
