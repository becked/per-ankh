C4 Architecture Model

Per-Ankh (per-ankh.app) modelled with the C4 visual notation — four nested levels of abstraction, each zooming one step further in: a System Context for the whole app and its neighbours, a Container view of the deployable runtimes and data stores, a Component view of the pieces inside the two Workers and the parser, and a note on the Code level. Browser-side parsing, a SvelteKit SSR Worker, a hand-rolled API Worker, and three Cloudflare storage bindings — Discord is the only third party in any production flow.

About this model

The C4 model describes software architecture at four levels of detail, like zooming a map. You stop descending once the diagram answers the question at hand. Per-Ankh is small enough that Levels 1–3 carry almost everything worth drawing; Level 4 is intentionally left to the code.

L1 · Context Per-Ankh as a single box, the people who use it, and the systems it depends on.
L2 · Containers The separately-deployable/runnable things: SSR Worker, API Worker, in-browser parser, D1, R2, KV.
L3 · Components The major modules inside each container and how they collaborate.
L4 · Code Classes/functions — not drawn; the TypeScript source is the source of truth.

How to read the diagrams

Colour and shape are consistent across every diagram below.

Person A human role. Tan border. Players, viewers, operators.
Container (this system) An orange-bordered runtime owned by Per-Ankh — a Worker, the browser app, or the parser.
Data store A Cloudflare binding — D1, R2, KV. Drawn as a cylinder.
External system A dependency Per-Ankh does not own — Discord OAuth, the Old World game itself.

Level 1 — System Context

Who uses Per-Ankh and what it talks to. The whole application is one box. Players generate save files by playing Old World on their own machine, upload them, and explore the parsed analytics; anonymous viewers follow public game and tournament links; operators run the admin surface. Discord is the single sign-on identity provider.

Level 2 — Containers

Zooming into the Per-Ankh box. Four runtime tiers plus one external service. Two tiers are Cloudflare Workers (SSR + API); a third "container" is the parser, which runs as a Web Worker inside the user's browser on the upload page; the storage tier is four sibling bindings on the same Cloudflare account (two D1, R2, KV). The legacy share viewer is a frozen static SvelteKit app sharing UI code via symlinks.

Level 3 — Components

Zooming inside the two Workers and the parser. These are the major modules, mapped to source directories so the diagram stays honest against the tree.

C1 Inside the API Worker

A single fetch handler (cloud/src/index.ts) dispatches to feature handlers by URL pattern. Cross-cutting concerns — session resolution, validation, scoping — sit between the router and the handlers; every handler talks to the storage bindings through the same three accessors.

C3 Inside the save parser

A pipeline that turns a raw Old World save ZIP into the structured game blob the app uploads and renders — all in the browser, off the main thread. Each domain parser under parsers/ extracts one slice of the XML; derived series are computed before validation gates the result against PARSER_VERSION.

Version contract: bumping PARSER_VERSION (src/lib/parser/types.ts) requires adding the string to KNOWN_PARSER_VERSIONS in the Worker, or uploads fail with INVALID_BLOB: Unknown parser_version. Deploy the Worker first.

Level 4 — Code

C4's deepest level drops to individual classes and functions. The model stops here deliberately: at Per-Ankh's size, a code-level diagram would duplicate the TypeScript and rot the moment a function is renamed. The source — and its tests — are the source of truth.

  • API handlers & routing: cloud/src/index.ts plus the per-feature modules listed in the C1 diagram.
  • Tournament engine: pairing, bracket, standings, and map selection under cloud/src/tournament/ carry their own *.test.ts — the densest logic in the codebase.
  • Parser pipeline: the domain parsers under src/lib/parser/parsers/.
  • Data shape: Valibot schemas in cloud/src/schemas/ and cloud/src/validation.ts.

Reference — data stores

The three Cloudflare bindings the API Worker owns. The browser never reaches them directly; every read and write is mediated by a Worker handler.

Binding Service Holds Notes
SHARE_DB D1 (SQLite) Relational metadata: users, games, events, collections, online_ids, and the tournament tables (tournaments → rounds → matches → slots). Forward-only numbered migrations in cloud/migrations/ (28 to date).
SECURITY_DB D1 (SQLite) Second D1 — the security-events drain (Skiff). Append-only security_events tee, isolated from app data. Own migrations in cloud/migrations-security/, applied out-of-band.
SHARE_BUCKET R2 (object store) Raw save ZIPs (saves/<id>.zip) and parsed, gzipped game blobs (games/<id>.json.gz). Keys are always server-constructed, never client-supplied.
SESSIONS_KV KV (key-value) Sessions (session:*, 30-day TTL) and in-flight OAuth state (oauth:*, 5-min TTL). The session cookie is an opaque KV key — no JWT.
