Section 3.4: Content-Hashed Atlas + Sprite Paths A walkthrough of the caching change in docs/cloud-deploy-plan.md

1. What we're solving

The map view fetches roughly 26 MB of pre-baked atlas images on first paint of /games/[id], plus a long tail of small sprite PNGs. We want browsers to cache these aggressively — ideally forever — because they don't change between page loads in a normal session, and re-downloading on every navigation is painful.

But we also re-bake atlases regularly as the Pinacotheca sprite-rendering pipeline iterates. So "cache forever" needs to coexist with "the file at this URL might change tomorrow." Those two goals fight each other unless we restructure the URL.

The core idea Make the URL itself encode the content. If terrain-3d.webp changes, its URL changes. A returning visitor whose browser cached the previous version still has it — at the old URL, which nothing references anymore, so it sits idle until the cache evicts it. The new HTML points at the new URL, which is a clean cache miss and gets fetched fresh. There is no "stale" state that needs to be invalidated.

2. HTTP caching primer

Skip if you already know this. Calling out the bits that matter for the rest of the doc.

Cache-Control response header

The server sends this on every response. It tells the browser (and intermediate caches like CDNs) how long the response is allowed to be stored and reused without re-asking the server.

max-age=N
Cache this response for N seconds. After N seconds, ask the server again before reusing it.
public
Anyone in the chain can cache it (browsers, CDNs, corporate proxies). Default for static assets.
private
Only the end-user's browser can cache it — intermediate caches must skip. For per-user data.
immutable
The thing at this URL will never change. Don't even bother revalidating when the user reloads the page. Strong promise; if you break it, users see stale content for the full max-age with no way to recover short of opening devtools.
must-revalidate
Once the response is stale (past max-age), you must ask the server before reusing it — can't serve stale-while-error.

The two cache-hit paths

There are actually two ways a "cached" response avoids a download:

  1. Fresh hit (free). The cached response is still within max-age — the browser uses it without talking to the server at all. Zero bytes, zero round-trips.
  2. Revalidation hit (cheap, but not free). The response is past max-age, but the browser has an ETag or Last-Modified from the prior response. It sends an If-None-Match request; the server replies with a 304 Not Modified and no body. One round-trip, no body bytes.

immutable short-circuits even (2). A 304 still costs a round-trip; immutable says "don't even ask."

Why immutable, max-age=1y is dangerous on a stable URL

If you set Cache-Control: immutable, max-age=31536000 on /atlases/terrain-3d.webp, and then re-bake and replace that file with new content, every browser that has ever loaded the old version will keep serving the old version for up to one year. Hard refresh works, but normal navigation doesn't — the browser never asks. There is no server-side workaround.

3. Why content-hashed URLs solve it

The fix is to make the filename a function of its content. If the content changes, the filename changes, and the old filename is no longer referenced by any HTML or JS the browser receives. So the browser's cache for the old URL is harmless — nothing points at it anymore — and the new URL is a clean cache miss that gets fetched fresh.

In-place URL + immutable (broken)
Content-hashed URL + immutable (works)

Vite/SvelteKit already does this for the JS bundle — everything under _app/immutable/ is content-hashed. We're extending the same pattern to the assets in static/ that Vite leaves alone.

4. On Cloudflare specifically

A few Cloudflare-specific bits worth knowing before we touch any code.

Workers Static Assets

When you deploy a SvelteKit app via @sveltejs/adapter-cloudflare 7.x, the adapter splits the output in two:

  • SSR Worker — the JS bundle that handles dynamic routes (everything not in static/). Lives at .svelte-kit/cloudflare/_worker.js.
  • Static asset bundle — everything in static/ plus Vite's hashed JS chunks. Cloudflare serves these directly from the edge — the request never invokes your Worker. They sit in front of the Worker in the request pipeline.
Request path on Cloudflare

This matters because _headers rules apply to the static asset path only. The SSR Worker can set its own headers in hooks.server.ts, but assets bypass it entirely. Good for us — serving 6 MB of atlas data through a Worker would burn CPU time we don't need to spend.

The _headers file

Inherited from Cloudflare Pages. Plain text, glob-pattern → header list. Workers Static Assets honors it natively (per the adapter-cloudflare docs). The format is whitespace-sensitive: 

# comments allowed
/atlases/*
  Cache-Control: public, max-age=31536000, immutable

/sprites/*
  Cache-Control: public, max-age=604800

Cloudflare's edge cache vs. browser cache

Cache-Control applies to both. The CF edge will cache per the same rules and serve subsequent requests without hitting your Worker. Browsers will cache locally. With immutable, max-age=1y, both layers cache for a year — so cold-start overhead happens once globally on first deploy of a hashed URL, then never.

6. The proposal

Hash both atlases and sprites. One mental model for static binary assets: each one is content-addressable, lives in a generated manifest, and gets served with Cache-Control: immutable, max-age=1y.

Asset class Treatment Cache-Control
Atlases (.webp + .json) hashed public, max-age=31536000, immutable
nation-asset-aliases.json hashed public, max-age=31536000, immutable
Sprites (.png) hashed public, max-age=31536000, immutable

Why hash everything (not just atlases)

Two reasons, both about code quality rather than caching mechanics:

Consistency

"Static binary assets are content-hashed and live in a manifest" is one rule. "Atlases are hashed but sprites aren't, sprites get a different cache policy" is two rules plus an arbitrary line between them. There is no principled basis for the line — sprites and atlases are both pre-baked binary outputs of the scripts/bake-* pipeline, both served as static assets, both invalidated by Pinacotheca-side changes. Treating them the same removes a category of decision (which class is this asset? what cache policy applies?) that we'd otherwise have to revisit every time a new asset class appears.

Hand-maintained code shrinks

Today, src/lib/game-detail/helpers.ts builds sprite URLs by string concatenation and gates them behind hardcoded existence sets:

  • KNOWN_TECH_SPRITES — ~60 entries enumerating "which tech sprites exist" (lines 388–450)
  • KNOWN_UNIT_SPRITES — ~90 entries doing the same for unit sprites (lines 453–543)

These exist purely as runtime guards: if a save references TECH_FOO and we don't ship art for it, we want to fall through to a fallback rather than 404. With a sprite manifest, the manifest is the existence check — both sets become redundant and can be deleted. Net change to helpers.ts: ~150 lines of hand-maintained hardcoded data go away, replaced by a generated module.

getSpritePath() simplifies to a manifest lookup with the typo-fix and tech-falls-back-to-unit logic preserved: 

function getSpritePath(category: SpriteCategory, enumValue: string): string | null {
  if (category === "techs") {
    const fixed = TECH_SPRITE_FIXES[enumValue] ?? enumValue;
    return SPRITE_MANIFEST[`techs/${fixed}`]
        ?? SPRITE_MANIFEST[`units/${fixed.replace(/^TECH_/, "UNIT_")}`]
        ?? null;
  }
  if (category === "crests") {
    return SPRITE_MANIFEST[`crests/CREST_${enumValue}`] ?? null;
  }
  return SPRITE_MANIFEST[`${category}/${enumValue}`] ?? null;
}

Follow-on consolidation

Once the sprite manifest is the source of truth for "what art ships," two more pieces become candidates for removal:

  • src/lib/generated/crests.ts exposes FAMILY_CRESTS and NATION_CRESTS as existence sets. These can be derived views over the sprite manifest (filter keys starting with crests/CREST_FAMILY_), or the existing callsites in SpriteMap.svelte can hit the manifest directly.
  • scripts/bake-crests.ts exists only to emit crests.ts. If crests.ts goes, so does its baker — scripts/bake-sprites.ts already walks the same directory tree to build the sprite manifest.

Not strictly part of §3.4, but worth doing in the same change since the unification opens the door. We'd be left with one bake step (bake-sprites.ts) and one manifest module covering everything under static/sprites/, plus the parallel atlas baker and atlas manifest.

One honest trade-off

Tighter coupling to bake correctness Today the KNOWN_*_SPRITES sets are a tripwire: if Pinacotheca starts rendering a new tech but you forget to re-bake, the runtime falls through cleanly because the hardcoded set says "no sprite." With manifest-as-truth, the bake script is the only safety check. A stale or partial bake produces a manifest with missing entries, and the runtime tries to load whatever is there. The fix is to make the bake fail loudly on missing sources rather than silently producing a partial manifest — not a real downside, but worth knowing.

7. Bake-script changes

Two bakers, two manifests

Two parallel structures, same shape:

  • Atlas baker. The existing scripts/lib/atlas-bake.ts + the three callers (bake-terrain-3d.ts, bake-improvements.ts, bake-resources.ts). Output: hashed .webp/.json pairs in static/atlases/, plus src/lib/generated/atlas-manifest.ts.
  • Sprite baker. scripts/bake-sprites.ts, which today copies PNGs from Pinacotheca into static/sprites/<category>/. Output: hashed .png files plus src/lib/generated/sprite-manifest.ts. Subsumes the existing bake-crests.ts (its directory scan over static/sprites/crests/ is already covered).

What to hash

Hash the output file's bytes, not the source. For atlases: sha256(webpBuf).slice(0,8). For sprites: same operation on the copied PNG buffer.

  • For atlases, the .json coordinates depend on bake logic (cell geometry, packing) as well as source pixels. Hashing the output catches changes to bake parameters automatically; a source-only hash would miss them.
  • sharp's lossless WebP and PNG encoders are deterministic, so identical input produces identical output hashes — re-bakes that don't change anything reuse old URLs (cache hit preserved).
  • For multi-source atlases like improvements-urban-EGYPT, "the source" isn't a single file anyway, so output-hashing is the only honest option.

Atlas writer

// scripts/lib/atlas-bake.ts
import { createHash } from "node:crypto";

export interface HashedAtlasOutput {
  logicalName: string;       // "terrain-3d"
  hash: string;              // "a1b2c3d4"
  webpFilename: string;      // "terrain-3d.a1b2c3d4.webp"
  jsonFilename: string;      // "terrain-3d.a1b2c3d4.json"
}

export async function writeAtlas(
  out: AtlasOutput,
): Promise<HashedAtlasOutput> {
  const webpBuf = await sharp({ /* ... */ })
    .composite(out.composites)
    .webp({ lossless: true })
    .toBuffer();

  const hash = createHash("sha256")
    .update(webpBuf)
    .digest("hex")
    .slice(0, 8);

  const webpName = `${out.name}.${hash}.webp`;
  const jsonName = `${out.name}.${hash}.json`;
  const manifestText = JSON.stringify(out.manifest, null, 2) + "\n";

  await writeFile(resolve(out.outputDir, webpName), webpBuf);
  await writeFile(resolve(out.outputDir, jsonName), manifestText);

  return { logicalName: out.name, hash, webpFilename: webpName, jsonFilename: jsonName };
}

Sprite writer

bake-sprites.ts walks static/sprites/<category>/ and copies each PNG through a hash-and-rename. Manifest keys are <category>/<basename> (without .png) so consumers in helpers.ts can construct lookup keys directly: 

// scripts/bake-sprites.ts (sketch)
async function hashAndCopy(
  src: string,
  destDir: string,
  basename: string,
): Promise<{ filename: string; hash: string }> {
  const buf = await readFile(src);
  const hash = createHash("sha256").update(buf).digest("hex").slice(0, 8);
  const filename = `${basename}.${hash}.png`;
  await writeFile(resolve(destDir, filename), buf);
  return { filename, hash };
}

// For each <category>:
//   for each PNG: hashAndCopy → record manifest[`${category}/${basename}`] = `/sprites/${category}/${filename}`
// Then write src/lib/generated/sprite-manifest.ts

Cleaning up stale outputs

When a hash changes, the old hashed file lingers under static/atlases/ or static/sprites/. Two layers of cleanup, both in place:

  1. Per-bake wipe. Each bake script clears its own output slice before writing. bake-improvements.ts clears improvements-* files; bake-sprites.ts clears static/sprites/<category>/*.png for each category it processes. Keeps single-bake invocations clean during development — running npm run bake:improvements alone leaves the rest of static/atlases/ untouched.
  2. Final reconcile. scripts/reconcile-assets.ts runs at the end of npm run bake:all. It diffs static/atlases/ ∪ static/sprites/**/* against the entries in atlas-manifest.ts and sprite-manifest.ts, and deletes any file not referenced by either manifest. Catches orphans from bake-script bugs, renamed assets whose old name no longer has an owner, and partial-bake leftovers.

Belt-and-suspenders. (1) keeps the dev workflow ergonomic; (2) keeps the deployed bundle clean and tolerant to bake-script mistakes. The reconcile is roughly thirty lines: list files, list manifest references, set-difference, rm.

9. The _headers file

Where it goes

The deploy plan says repo root. Worth verifying against adapter-cloudflare 7.x docs before committing — some adapter versions copy from static/_headers, others from ./_headers. (Quick check: build with npm run build, look at what ends up in .svelte-kit/cloudflare/.)

What it should contain

Both classes are content-hashed and get the same rule. One _headers rule, one mental model: 

# All baked binary assets are content-hashed → safe to cache forever.
/atlases/*
  Cache-Control: public, max-age=31536000, immutable

/sprites/*
  Cache-Control: public, max-age=31536000, immutable

The legacy share viewer has its own _headers with security rules. Add the same two cache rules there:

  X-Content-Type-Options: nosniff

/atlases/*
  Cache-Control: public, max-age=31536000, immutable

10. Decisions

All five resolved — recorded here for the record.

D1. Hash scope — decided Hash everything: atlases, the alias JSON, and all sprites. One rule for static binary assets, one cache policy, one manifest pattern. The KNOWN_TECH_SPRITES / KNOWN_UNIT_SPRITES hardcoded sets in helpers.ts are deleted, and the generated/crests.ts existence sets fold into the sprite manifest. See §6 for rationale.
D2. Hash input — decided Hash output bytes, truncated to 8 hex chars. sha256(webpBuf).slice(0,8) for atlases; sha256(pngBuf).slice(0,8) for sprites. Output-hashing catches bake-script changes (e.g., a tweak to SAFE_SCALE in atlas-bake.ts) that a source-hash would miss, and is the only honest option for multi-source urban atlases. sharp's lossless WebP and PNG encoders are deterministic, so identical input produces identical hashes — cache hits preserved on no-op re-bakes.
D3. Stale-output cleanup — decided Hybrid: per-bake wipe + final reconcile. Each bake clears its own output slice before writing (keeps single-bake invocations clean during dev), and a scripts/reconcile-assets.ts step at the end of npm run bake:all diffs on-disk files against both manifests and deletes orphans (catches bugs in individual bake scripts and renamed-asset leftovers). See §7 for details.
D4. _headers location — decided Project root. Verified by reading node_modules/@sveltejs/adapter-cloudflare/index.js: line 32 throws if _headers is found in static/; line 139 reads it from ./_headers at the build cwd. The adapter then appends its own auto-generated rules for /_app/immutable/* (which already get public, immutable, max-age=31536000) to whatever you wrote — our /atlases/* and /sprites/* rules sit alongside those.
D5. web/static/_headers parity — decided Yes, add the matching cache rules to web/static/_headers. The legacy viewer is dying but not dead — deploy plan §3.8 redirects old /share/[id] URLs to it indefinitely, and it consumes the same atlases and sprites via symlink. Without parity, those requests get Cloudflare Pages' default caching, which gives up the "cache forever" property on shared assets. Four-line append; just do it.
