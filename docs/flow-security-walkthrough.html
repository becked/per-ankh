Action Flow Security Walkthrough

Twelve action flows through Per-Ankh's Browser → SSR Worker → API Worker → D1 / R2 / KV topology, each rendered as a sequence diagram with a step-by-step security commentary. The first six sections frame the system as a whole — boundaries, auth model, validation layers — so the per-flow cards can be terse. Every concrete claim is cited with file:line against the working tree at 2026-05-10.

1. System overview

Four runtime tiers and one external service. Two of the tiers (SSR Worker, API Worker) are Cloudflare Workers; the storage tier is three sibling bindings on the same Cloudflare account. Discord is the only third-party in any production flow.

2. Trust boundaries

What's trusted at each tier, and what each tier is allowed to assume about the layer above it.

Tier Trusted to do Never trusted to do
Browser Render UI, run the Web Worker parser for upload UX, hold the session cookie (HttpOnly). Authoritatively answer "is this game public", "is this user the owner", "is this file already on the server" — every such question is re-asked of the API server-side.
SSR Worker
(per-ankh.app)		 Render HTML, apply security headers (XCTO, XFO=DENY, Referrer-Policy, Permissions-Policy, Report-To), forward the incoming Cookie header on cross-origin API fetches. Make security decisions on behalf of the API — it has no D1/R2/KV bindings and no Discord secret. The SSR Worker is a renderer with extra plumbing, not a security boundary.
API Worker
(api.per-ankh.app)		 Validate every request (Valibot + bound-checks), enforce ownership against D1, run rate limits, decompress with caps, hash uploads server-side, mint sessions, strip PII for public reads, emit structured logs with PII deny-list. Reject failure on the response path (audit-event inserts are fire-and-forget so a logging hiccup can't 500 a real request), or surface internal error details (stack traces, internal IDs) in error responses.
D1 / R2 / KV Persist data exactly as the Worker stores it; enforce constraints (UNIQUE, FK, NOT NULL) the schema declares; honour TTLs on KV keys. Be reached by anything other than the API Worker. No public-read R2 buckets, no D1 HTTP API exposed.
Discord OAuth Authenticate the user, enforce its own redirect_uri allowlist, return signed identity claims, mint single-use authorization codes. Be the only authorization gate — the Worker re-checks the username allowlist locally on every login and every /v1/auth/me.
The critical boundary is between the browser and the API Worker. Everything in the SvelteKit app is client-controllable — CSP, safeNext(), the Web Worker parser, even the SSR-rendered HTML — and the API Worker is responsible for validating every assertion that crosses it. The SSR Worker's job is presentation + cookie forwarding; treating it as a security control would create a gap.

3. Authentication model

Discord OAuth 2.0 with PKCE, gated by a server-side username allowlist, session token in KV with a 30-day TTL. Three properties are load-bearing:

  • State + PKCE bind the OAuth round-trip to the browser tab. 32-byte state and 32-byte verifier (CSPRNG) live in KV under a 5-minute key. The KV id is in an oauth_pending cookie that the browser presents at the callback. The KV entry is deleted unconditionally on the first callback read — a stolen authorization code can't be replayed.
  • The username allowlist is re-checked every request, not just at login. handleMe compares the lowercased Discord username on the session against the current ALLOWED_DISCORD_USERNAMES value. Revoking access is a secret update + a session-TTL wait, not a database migration.
  • The session token is opaque and KV-backed. 32-byte nanoid, server-side revocable, no claims encoded client-side. There is no JWT to forge.

4. Authorization & ownership

Per-resource ownership is the only authorization primitive — no roles, no admin layer in the request path (admin tooling sits behind wrangler CLI access, not a web endpoint).

  • Every authed handler resolves session.user_id from KV once, then filters all D1 reads/writes on it. Reads either include WHERE user_id = ? directly (stats, games list, collections list) or do a row lookup followed by an equality check (game detail, patch, delete).
  • Mutations on private resources collapse "doesn't exist" and "isn't yours" into a single 404. See games.ts:1328-1330 (PATCH) and games.ts:1524-1527 (download). The handleDelete for legacy share is the exception and is tracked as F-17 / Info in the review.
  • Cross-resource ownership is re-checked. Moving a game into a collection (PATCH collection_id) requires the collection to also be owned by session.user_id — see games.ts:1371-1379.
  • Public reads strip PII, never grant write. is_public = TRUE is the only flag that ever opens a resource to anonymous read; it never grants delete, download, or patch. See games.ts:1199-1208.

Five validation layers in front of every write path. The pipeline is strictly server-side: nothing the client tells the server is taken on trust.

6. Storage & secrets

Three bindings, all server-constructed keys, no public-read flags.

D1 — SHARE_DB All queries via db.prepare(sql).bind(...); no string concatenation into SQL. event_type is a fixed allowlist used as a literal (not a bind param) by design — closes the only structural SQL injection vector.
R2 — SHARE_BUCKET Keys are always games/<nanoid>.json.gz or saves/<nanoid>.zip. Content-Type + Content-Encoding fixed server-side. No signed URLs; all reads go through the API Worker (so it can apply auth, PII strip, rate limit).
KV — SESSIONS_KV oauth:<nanoid> (5min TTL), session:<nanoid> (30d TTL). TTLs are the sole expiry mechanism — no separate cleanup job.
Secrets DISCORD_CLIENT_SECRET and ALLOWED_DISCORD_USERNAMES set via wrangler secret put only. Never in wrangler.toml, never in .env committed to git.

Walkthrough

  1. 1-2. Frontend sanitizes ?next= via safeNext() (rejects non-/, //, backslash) before sending — server re-validates on receipt, so this is purely UX.
  2. 3-4. Worker mints state, PKCE verifier, KV id (nanoid). All randomness from crypto.getRandomValues(). KV write has a 5-minute TTL — no separate cleanup needed.
  3. 5. authorize_url includes the SHA-256 PKCE challenge (S256), state, and prompt=none so returning users skip the consent screen. oauth_pending cookie carries only the KV id — the verifier and state never leave the server.
  4. 6-8. Browser navigates to Discord. Discord enforces its own redirect_uri allowlist — see F-10 for the defense-in-depth gap (Worker doesn't re-check). Authorization code returned via redirect.
  5. 9-11. Callback reads the KV entry and deletes it before doing anything else. A leaked code can't be replayed because the corresponding state has already evaporated.
  6. 12. State comparison is timing-safe via crypto.subtle.timingSafeEqual. redirect_uri match is plain equality (not a secret).
  7. 13-16. Token exchange uses the stored verifier — PKCE prevents code-interception even if the redirect chain were tampered with. Discord profile fetched with the access token; the token is never stored or returned to the client.
  8. 17. Username allowlist re-checked here (and again at every /v1/auth/me — F2 step 5). Empty / unset secret fails closed.
  9. 18-22. User row upserted (ON CONFLICT(discord_id) so the same Discord user always maps to the same user_id). Personal collection seeded once. Session token minted, KV write before the cookie response.
  10. 23. Worker returns the server-validated next; client re-runs safeNext() as belt-and-suspenders in case an older OAuthPending shape lacks the field.
Trust transitions: (a) Browser → Worker at step 2 — Worker treats redirect_uri and next as untrusted strings, stores them only after safeNext() applies. (b) Worker → Discord at step 13 — Worker presents only public client_id + server-held client_secret + PKCE verifier; access token never round-trips to the client. (c) Discord → Worker at steps 14, 16 — Worker treats the profile as identity claims to be re-checked against the local allowlist before any session is minted.
If the state check (step 12) is bypassed: an attacker can complete the OAuth callback for a victim using a code they obtained out-of-band, hijacking the session that gets minted. The KV delete-on-read makes this a single-shot attack; the timing-safe compare prevents oracle attacks.
If the allowlist check (step 17) is bypassed: any Discord user becomes an authenticated app user. The check is the entire beta gate.

sequenceDiagram autonumber participant B as Browser participant API as API Worker participant KV as KV participant D1 as D1 B->>API: GET /v1/auth/me

request_id := crypto.randomUUID() API->>API: parseCookies(Cookie)

Walkthrough

  1. 1-2. Every request enters via runWithLogContext which sets up an AsyncLocalStorage frame with a fresh UUID request_id, the route pattern, and a place for the resolved user_id.
  2. 3. Cookie parsed by a small helper that ignores malformed entries — no third-party cookie library on this hot path.
  3. 4-7. KV miss is indistinguishable from "expired"; KV's own TTL handles cleanup. Pre-allowlist-migration sessions lack discord_username and are treated as missing — forces a clean re-login when the gate changes.
  4. 8. setUserId annotates the request-scoped log context so the access log emitted at the end of the request carries the resolved user_id without any handler having to pass it through.
  5. 9-12. /v1/auth/me specifically: re-checks against the current allowlist on every call. If the username has been removed from ALLOWED_DISCORD_USERNAMES since login, the session is destroyed here. Other authed endpoints don't repeat this check — they trust the allowlist verdict at login time within the session TTL window.
Trust transition: the cookie crosses the browser-to-Worker boundary in plaintext at the HTTPS layer. HttpOnly prevents JS from reading it; SameSite=Lax prevents cross-site sub-resource attachment; Secure prevents plain-HTTP transmission. The token itself is opaque (no claims), so capture is the only attack — covered by the cookie attributes above plus rotation on logout.
If the KV lookup is bypassed: an attacker who has guessed a session token gains the victim's identity. nanoid(32) = ~256 bits, so guessing is infeasible; capture is the only realistic avenue. If the allowlist re-check is bypassed: users whose access was revoked retain their session until KV expires it (max 30 days).

Walkthrough

  1. 1. Client POSTs with credentials: 'include'; CORS preflight against allowlist + SameSite=Lax blocks cross-site cookie attachment on POST forms.
  2. 2-4. KV entry deleted; audit row written only if a real session was torn down. Anonymous POSTs still return 204 — the endpoint is idempotent so retries from disconnected clients don't fail.
  3. 5. Clear cookie must use the same Domain attribute or the browser treats it as a different cookie and won't actually clear the real one.
  4. 6. Frontend does a window.location.assign('/') hard reload to flush any stale client-side state — see commit f295d20 ("hard-reload on logout to avoid stuck state").
Trust transition: the only state change is the KV delete. R2 and D1 user data are not touched (those need F-01's account-deletion endpoint). The audit-log insert is best-effort via try/catch — a D1 hiccup must not prevent the cookie from being cleared.
If the KV delete fails silently: the cookie is cleared but the session token remains valid in KV. If the user's cookie was captured before logout, the attacker can continue using it until the 30-day TTL expires. Today this is a tolerated degraded state; a per-user session index would let future revocation guarantee both sides.

Walkthrough

  1. 1-2. SSR route loads. handleFetch sees no Cookie header and doesn't forward anything.
  2. 3. Cross-origin fetch from SSR (per-ankh.app) to API (api.per-ankh.app). Both are Cloudflare-internal but they're separate Workers — the request still goes through normal CORS gates.
  3. 4-5. Worker resolves session = null. D1 lookup pulls the bare access-control flags (no blob yet — cheap if we're going to deny).
  4. 6-7. Existence isn't disclosed for private games: anonymous + private returns 401 (frontend bounces to login), signed-in + private + non-owner returns 403 (covered in F7). Public path continues.
  5. 8-9. Scraper UA exemption — Discord/Slack/Twitter etc. skip rate limiting and the audit log. F-03 in the security review flags both the spoofability and the audit-log gap.
  6. 10. Rate-limit check via D1 events table — global per-IP (not per-POP), so a determined scraper can't shard across Cloudflare colos.
  7. 11-13. R2 fetch, decompress with cap, JSON parse, recursive PII strip. The deep walk catches online_id wherever the schema may put it in the future, not just in player_roster.

Trust transitions: the only data crossing back to the browser is the PII-stripped blob — every online_id field is rewritten to null before the response is stringified. The recursive walk is the only line of defense; a future parser version that adds a new online_id location is caught without any code change here.
If stripOnlineIds misses a field: platform IDs (Steam/GOG/Epic) for opponents leak to anonymous viewers — the same opponents who didn't choose to upload the save are exposed. If the is_public gate is bypassed: a private game's blob (including full PII) becomes anonymously readable. The gate is a single integer column comparison; rate-limit pressure cannot bypass it.

Walkthrough

  2. 2. handle() hook attaches security headers to every SSR response.
  3. 3-4. +page.ts fires three parallel fetches via cloudApi.*({fetch}). handleFetch recognizes api.per-ankh.app (or localhost:8787 in dev) and copies the incoming Cookie header onto the outbound request — without this, hard refreshes of authed pages would always log out.
  4. 5-13. Three parallel API calls, each independently authed via the same session token. Each handler filters D1 on session.user_id; no cross-user data fan-in is possible.
  5. 14. filterSerializedResponseHeaders allowlists exactly three headers when SSR forwards response metadata to the client — most importantly, Set-Cookie and other sensitive headers from the API response can't leak through.

The allowlist is the only thing between an API Set-Cookie and the client; treat it as security-critical.
If the response-header allowlist is widened: Set-Cookie from API responses could leak into the SSR response and clobber the client's cookies (or in adverse configurations, set cross-origin cookies). Current allowlist is three specific headers — extend with care. If handleFetch forwards Cookie to non-API URLs: any same-page server-side fetch (analytics, third-party APIs) would leak the session token to a third party. The check is currently an exact-hostname match.

Walkthrough

  1. 1-3. Auth required even for download of public games — the raw ZIP contains the original OnlineIDs that the JSON path strips. Authed-only download gives a per-user audit trail.
  2. 4-7. 404 for private non-owner — existence is not disclosed (matches F7 and F10 patterns).
  3. 8-11. Two rate limits in series: per-user (50/h) is the meaningful one since auth is required; per-IP (100/h) catches one user spamming from one IP without bypassing the per-user limit.
  4. 12-14. Filename sanitization is the only place user-supplied data (the game name) flows into a response header. Path separators, control chars, drive colons, quotes are all stripped. RFC 6266 + 5987 emit both an ASCII fallback and a UTF-8 form.
  5. 15-16. R2 body streamed straight through — no buffering, memory-safe at the 50MB R2 ceiling. Access-Control-Expose-Headers lets cross-origin JS read the Content-Disposition so the browser uses the right filename.
Trust transition: the game_name string is the only user-controlled value that lands in a response header. buildContentDisposition double-encodes — ASCII for older clients (latin-1 only), URL-encoded UTF-8 for modern ones. CR/LF and quote characters are stripped before the ASCII form is emitted, so header injection via a crafted game name is not possible.
If the filename sanitizer misses CR/LF: Content-Disposition header injection — attacker could split the response into an arbitrary header set. Today the strip pattern covers [\x00-\x1f\x7f]. If auth is bypassed: raw ZIPs (with full PII) become anonymously downloadable from any public game URL, undoing the stripOnlineIds protection.

Walkthrough

  1. 1-5. Browser-side parse runs in a Web Worker for UX (lets the main thread stay responsive). The parser enforces its own ZIP-bomb caps: 50MB compressed, 100MB uncompressed, 10 entries max, 100x compression ratio. fast-xml-parser is configured with parseAttributeValue: false, parseTagValue: false, ignoreDeclaration: true — no XXE surface.
  2. 6-7. Client gzips the JSON; raw ZIP gets transferred (zero-copy) so it can be re-uploaded to the server. The client's parse is for UX only — the server validates everything fresh.
  3. 8. Multipart POST with three parts. credentials: 'include' attaches the session cookie.
  4. 9-11. Three rate-limit checks in series (per-user, per-IP, global). Uploads and re-imports share the same budget — both do the same R2/D1 work.
  5. 12-13. Server-side size caps re-checked even though the client claims to have respected them. The 50MB streaming decompression cap is enforced inside the decode loop, not after.
  6. 14. Valibot schema validates the JSON shape: known parser_version, array length bounds (e.g. player_history ≤ 20, event_logs ≤ 50000, map_tiles ≤ 50000), game_over: true, integer ranges for turn counts. The envelope is looseObject for forward-compat — extra keys preserved without affecting prototype semantics (JSON.parse never sets [[Prototype]]).
  7. 15. uploader_player_index validated separately: either null (observer mode) or a non-negative integer that resolves to a human player in the roster.
  8. 16. File hash is computed server-side. The client's idea of the hash never enters the trust path — dedup is enforced on raw-ZIP bytes the server actually received.
  9. 17-21. Dedup branches: identical-or-older parser_version → 409 (no point overwriting); newer parser_version → re-import (reuse game_id, REPLACE games row with FK cascade clearing children); new → fresh nanoid + Personal collection default.
  10. 22-24. R2 puts in parallel. Content-Type and Content-Encoding are server-set; the user can't influence the stored MIME type. Custom metadata records user_id + created_at for ops/debug.
  11. 25-29. D1 batch is a single transaction. On failure for a new upload (not re-import), the just-written R2 blobs are deleted — avoids orphans. Re-import failures leave R2 alone because the existing row's blob has already been overwritten with the new payload (degraded but acceptable: the row still resolves to fresher data with a stale parser_version badge).
  12. 30-32. Online-ID capture is opt-in by virtue of the uploader's picker choice. Observer-mode uploads (uploaderIndex === null) skip this entirely — the uploader isn't claiming any player.
Trust transitions: three places where untrusted bytes cross the boundary — (a) the gzipped data blob (decompressed with cap, JSON-parsed, schema-validated); (b) the raw ZIP (SHA-256'd server-side, then handed to R2 as-is); (c) the uploader_player_index integer (range + roster membership checked). Nothing the client says about the file (claimed hash, claimed parser_version, claimed player names) is taken on trust without server re-derivation or schema validation.
If the decompression cap is bypassed: a gzip bomb could exhaust Worker memory. Defended at three layers — client Web Worker, server decompressWithLimit (streaming cap inside the decode loop), and the upstream MAX_BLOB_COMPRESSED ceiling.
If the schema validator is bypassed: array-length attacks (e.g. event_logs: 10_000_000) would force unbounded D1 writes. Each array has an explicit v.maxLength.
If the dedup check is bypassed: users can fill their quota with duplicate uploads. Storage cost; no security impact.
If the ownership of game_id is wrong on a re-import: one user could overwrite another user's game blob. The dedup query is scoped WHERE user_id = ? so this is structurally prevented.

Walkthrough

  2. 3. Session required; body is JSON.
  3. 4-7. Ownership check first. 404 (not 403) is used deliberately — a signed-in user probing for someone else's game_id can't distinguish "doesn't exist" from "isn't yours."
  4. 8. Schema requires at least one of the two fields. collection_id integer must be ≥ 1 (or null to uncategorize).
  5. 9-11. Visibility-change rate limit only applies when is_public is actually in the body. A pure collection move doesn't burn the user's toggle budget.
  6. 12-15. Cross-resource ownership check: the target collection must also belong to the same user. Without this, an attacker could move their game into another user's collection (effectively "publishing" into someone else's space).
  7. 16-19. Two independent updates, each with its own audit event. The audit inserts are fire-and-forget — UPDATE success is the user-facing signal.
Trust transitions: two ownership checks per request when both fields are set — the game itself and the target collection. Both are WHERE … AND user_id = ? against session.user_id. The audit-event inserts use parameterized binds for the metadata JSON, which is built from server-known values (game_id, is_public/collection_id, ip).
If the cross-resource ownership check is bypassed: users could move their games into other users' collections — either a privacy issue (the target user sees content they didn't invite) or a denial-of-service tool (filling another user's collections with garbage). If 404 vs 403 disclosure leaks: attacker enumerates valid game_ids by status code. Currently both branches return identical 404 bodies.

Walkthrough

  2. 4-7. The 403 here (vs the 404 in PATCH/download) is the inconsistency flagged as F-17 in the security review — minor enumeration oracle, low impact because game_ids are 126-bit nanoids.
  3. 8-10. R2 deleted first (parallel). Cleanup errors are logged but don't fail the request — the alternative would leave the D1 row in place with no blob to serve, which is worse UX. Today an R2 failure can orphan blob bytes; this is a tolerated degraded mode.

If the ownership check is bypassed: any signed-in user can delete any other user's game. The check is a single row.user_id !== userId comparison.
If the R2 delete fails silently: orphan blob bytes accumulate. Storage cost only; no privacy or correctness impact, since the D1 row is gone so the blob is unreferenced and unreachable via the Worker.

Walkthrough

  2. VIEW (14-18). Public read. Per-IP rate limit via Cache API (per-POP counter, lightweight). Worker streams the decompressed JSON because the Cloudflare edge strips Content-Encoding from Worker responses — gzipped wire bytes would not be useful to the browser at this layer. Single-origin CORS for legacy.per-ankh.app (the frozen SPA).

Trust transitions: the entire legacy surface is capability-based — possession of the X-App-Key (for upload) or both X-App-Key + X-Delete-Token (for delete) is the authorization primitive. No session, no user model. R2 keys are <share_id>.json.gz with the share_id matched against a regex at the dispatch layer.
If timing-safe compare leaks: attacker could extract a valid delete_token via timing oracle, given a target share_id. Both compares use crypto.subtle.timingSafeEqual. If the legacy upload validator misses an array bound: the same payload-size attacks the cloud rewrite is hardened against — addressed by validation.ts's explicit array-length caps.
F-17 specifically: the 404-vs-403 split in delete enumerates valid share_ids; mitigated by 126-bit ID entropy.
