PR #49 — Tournament Feature Code Review

A code review of PR #49 (feat: tournament feature (private beta)) — a two-phase tournament feature (Swiss qualifier → single-elim championship) gated behind a private-beta allowlist. The change touches 100 files, +20,727 / −420 lines, and ships migrations 0006–0013, a new cloud/src/tournament/ Worker module, the SvelteKit src/lib/tournament/ UI, a tournament admin CLI under scripts/admin/commands/tournament.ts, and a new map-options bake script. This review focuses on correctness, project-convention adherence, performance, test coverage, and code organization — the security audit was done separately (2026-05-17 tournament security review) and found no vulnerabilities. Reviewed against tournament @ 13b3f08.

Executive summary

This PR is in good shape to merge. The pure-function algorithms (pairing, standings, bracket, maps, RNG) are isolated from D1, have direct unit tests, and use a deterministic mulberry32 seed so the integration tests stay reproducible. The Worker authorization layer is small, consistently applied through a handful of named wrappers (authedTournament, requireTournamentBeta, requireTournamentAdmin), and the integration-test sweep in cloud/test/integration/tournament/ already pins the contract on every state-changing endpoint. Migration design — triggers over create-new-table for adding constraints, explicit match_index column for deterministic ordering, BEFORE DELETE trigger paired with a worker-side 409 — is thoughtful.

Most findings below are polish rather than blockers: dead branches, an inconsistent default value across two patch surfaces, a handful of a11y gaps, modest duplication in optimistic-patch components, and one side-effect-ordering issue in the championship-transition handler that's not a security problem but could surprise an admin who submits a malformed body. There are no Critical or High findings.

The largest organizational tension is cloud/src/tournament/admin.ts at 2,012 lines. It still reads cleanly because each handler is well-commented and the helpers are factored, but it's the obvious next refactor when a future feature lands on the tournament surface.

R-01 Medium State-changing side effect runs before the request body is validated
Component
cloud/src/tournament/admin.ts — handleTransitionChampionship
Evidence
admin.ts:1411-1424

What happens

// admin.ts:1411
const rounds = await loadRounds(env, tournamentId);
const swissRounds = rounds.filter((r) => r.phase === "swiss");
const matches = await loadMatches(env, tournamentId);
for (const r of swissRounds) {
  const closeResult = await autoCloseRoundIfReady(env, r, matches);  // UPDATEs round.status
  if (!closeResult.ok) return errorResponse(...);
}

const body = await parseJsonBody(request, TransitionChampionshipSchema, cors);
if (!body.ok) return body.response;   // 400, but Swiss rounds already flipped to 'complete'

Why it matters

autoCloseRoundIfReady writes tournament_rounds.status = 'complete' for any Swiss round whose matches are all decided, then the handler parses the body. If the body fails validation (e.g. a malformed override_ranks), the request 400s but the round closures persist. The state change isn't wrong — the rounds were genuinely ready to close — but it's a surprising observable effect of a "failed" request and complicates manual recovery: an admin who fixes their override payload and retries gets a different system state than they started with.

Recommendation

Move parseJsonBody ahead of the auto-close loop. Body parsing is cheap and has no side effect, so doing it first preserves the invariant "if the response is a 4xx the DB is unchanged."

R-02 Medium swiss_advance_count default drifts between admin panels
Component
TournamentSettingsForm.svelte · TournamentConfigurationPanel.svelte
Evidence
TournamentSettingsForm.svelte:41 uses ?? 0; TournamentConfigurationPanel.svelte:25,108,112 uses ?? 1

What happens

The configuration panel and the settings form are two different editing surfaces for the same tournament field. When the server returns swiss_advance_count: null (during the setup phase), the two surfaces display different default values (0 vs 1). Whichever panel the admin uses last "wins" — the saved value depends on UI history rather than intent.

Recommendation

Pick one default. The schema comment in cloud/migrations/0006_tournaments.sql:37 says swiss_advance_count is null until start-swiss computes a suggestion via advanceCountSuggestion. The honest UX is "(not yet set — will default to N when you start the Swiss phase)" rather than picking 0 or 1 silently. If you must pick a number, use advanceCountSuggestion(divSlots) so the placeholder matches what start-swiss will actually do.

R-03 Medium Modal dialogs missing focus trap and return-focus on close
Component
MatchModal.svelte · TournamentSettingsModal.svelte · TournamentCreateModal.svelte
Evidence
MatchModal.svelte:187-200 · TournamentSettingsModal.svelte:22-35 · TournamentCreateModal.svelte:110-123

What's missing

All three modals correctly set role="dialog" aria-modal="true" tabindex="-1", but Tab can still escape to the page underneath, and on close focus doesn't return to the invoking trigger. The use:autofocus action focuses the first form field on open (good), but doesn't establish a trap.

Recommendation

Roll a small focus-trap helper (first/last sentinel tabindex=0 nodes that wrap focus on focusin) or bring in svelte-focus-trap. Store the previously-focused element when the modal opens and restore it on close. Same pattern will cover all three modals.

R-04 Medium Championship bracket SVG lacks screen-reader labels and list semantics
Component
ChampionshipBracketTree.svelte
Evidence
ChampionshipBracketTree.svelte:138-197

What's missing

The connector <svg aria-hidden="true"> is right — connectors are decorative. The matches themselves render as absolutely-positioned <a class="match"> nodes inside an unlabeled position:relative canvas with no round-column landmarks, no aria-label on each match link, and no enclosing list semantics. Tab order follows DOM order (round-1 first, then round-2, …) rather than visual flow, which is actually fine; it's the missing labels that hurt.

Recommendation

Add aria-label="Championship bracket" to the outer container, and per-match labels like aria-label="Round {n}: {a} vs {b}, {status}, winner {w}". The drag-and-drop slot reorder in +page.svelte:594-657 is also keyboard-inaccessible — worth flagging here even though it's outside this component, since the same admin audience uses both surfaces.

R-05 Low Dead "in_progress" branch in statusBadgeClass
Component
MatchModal.svelte
Evidence
MatchModal.svelte:171-175 (handler) · api-cloud.ts:834 (type)

Why it's dead

statusBadgeClass handles "in_progress" as a possible match.status, but TournamentMatch.status in api-cloud.ts is typed "pending" | "complete" | "forfeit" | "bye". The "in_progress" state exists on BracketRound.status (rounds, not matches). The branch never runs.

Recommendation

Tighten the parameter type (status: TournamentMatch["status"]) and drop the dead branch. TypeScript will then surface any future call site that tries to pass a round status to a match-status formatter.

R-06 Low Exported defaultsForScript has no callers
Component
src/lib/tournament/map-script-options.ts
Evidence
map-script-options.ts:70-79; grep -rn defaultsForScript matches only the definition.

Why it's dead

The JSDoc speculates a use ("optimistic add between save and invalidateAll"), but TournamentMapsPanel.commit() always awaits invalidateAll() directly after the patch — no optimistic intermediate state exists. Per the project's "don't design for hypothetical future requirements" guidance in CLAUDE.md, drop the function until a caller materializes.

R-07 Low getTournamentMatches drops legitimate falsy params
Component
src/lib/api-cloud.ts
Evidence
api-cloud.ts:440-443

What's wrong

The query-string builder filters with if (v), which drops empty strings, 0, and false. Today all params happen to be opaque IDs / enums so the bug is latent, but per the project rule in CLAUDE.md ("data layer (strict): use ?? for null/undefined; never use || for data computation"), this should be if (v != null) to pre-empt the next param that legitimately accepts "" or 0.

R-08 Low Four near-identical optimistic-patch commit() implementations
Component
TournamentMapsPanel.svelte · MapScriptOptionsBlock.svelte · TournamentConfigurationPanel.svelte · TournamentOverviewPanel.svelte
Evidence
TournamentMapsPanel.svelte:45-62 · MapScriptOptionsBlock.svelte:44-61 · TournamentConfigurationPanel.svelte:42-56 · TournamentOverviewPanel.svelte:32-46

What's duplicated

Each component implements the same status-discriminated union ({ kind: 'idle' | 'saving' | 'error' }), the same ApiError-narrowed message extraction, and the same invalidateAll() follow-up. The implementations drift: MapScriptOptionsBlock and TournamentMapsPanel roll back local state on error; TournamentConfigurationPanel and TournamentOverviewPanel do not. A future patch surface (the eighth, the ninth) inherits whichever drift is current.

Recommendation

Extract useTournamentPatch(tournamentId) returning { status, commit } to $lib/tournament/usePatch.ts. The rollback behavior stays in sync, and the noise drops from ~50 lines to ~5 per panel.

R-09 Low O(N) round-trips in downstreamBlocked
Component
cloud/src/tournament/admin.ts — downstreamBlocked
Evidence
admin.ts:1348-1375

What it costs

downstreamBlocked iterates rounds strictly after the retro-edited round and issues a SELECT COUNT(*) FROM tournament_matches WHERE round_id = ? per round. With M rounds × 2 divisions × R matches, this is M−1 round trips per retro-edit. Not hot today (retro-edits are rare), but a single grouped query SELECT round_id, COUNT(*) … WHERE round_id IN (?, ?, …) AND status != 'pending' GROUP BY round_id collapses it to one — fewer round-trips reduces the Cloudflare-Worker-to-D1 latency window in which a concurrent report could race.

R-10 Low Brittle "was a new row inserted?" detection in beta-grant
Component
scripts/admin/commands/tournament.ts
Evidence
scripts/admin/commands/tournament.ts:541

Why it's brittle

The handler does an INSERT … ON CONFLICT … DO UPDATE with RETURNING (CASE WHEN granted_at = datetime('now') THEN 1 ELSE 0 END) to discriminate inserted vs already-present rows. Correctness relies on (a) the column default also being datetime('now') and (b) SQLite evaluating both invocations within the same statement at the same second-precision boundary. This usually works, but it's coupled to a column-default coincidence rather than statement semantics. The DO UPDATE branch deliberately doesn't update granted_at to make the discrimination work.

Recommendation

Either pre-check existence with a separate SELECT 1 (one extra round-trip, much clearer), or add a sentinel column (e.g. last_seen_at) that the DO UPDATE bumps so RETURNING granted_at = last_seen_at becomes the signal.

R-11 Low Observer-mode upload page trusts URL params without admin pre-check
Component
src/routes/upload/+page.svelte · src/lib/BulkUploadModal.svelte
Evidence
upload/+page.svelte:25-29 (observerMode derived from ?observer=1) · enforcement at cloud/src/games.ts:1020-1036

What happens

observerMode is derived from the URL query string with no admin check. A beta user (non-admin participant) who lands on /upload?tournament_match_id=X&observer=1 gets the observer UI: extra slot-mapping pickers, a different "Player as" labelling. The Worker correctly rejects this (games.ts:1029 → 403 NOT_MATCH_PARTICIPANT), so this is a UX issue, not a security one — but the user only finds out after they've parsed the save and the upload submits.

Recommendation

On onMount, if observerMode && !user-is-admin-for-the-tournament, either redirect to the same URL without ?observer=1 or show a "you're not an admin for this tournament" message before unlocking the modal. Cheap to add since the page already fetches the match + standings for label resolution.

R-13 Info Bye wins indirectly enrich opponents' Solkoff / Buchholz
Component
cloud/src/tournament/standings.ts
Evidence
standings.ts:50-66

What's happening

collectOpponents correctly excludes byes from the opponent list (a player who got a bye doesn't have a "bye-opponent" inflating their Buchholz). But computeRecord counts a bye as a win for the receiver. So if player A receives a bye in round 1 and plays player B in round 2, then in B's Buchholz, A's wins (which now include the bye win) raise B's tiebreaker total. Standard Swiss has split conventions on this — some implementations strip bye wins from the strength-of-schedule pool, some don't.

Recommendation

Document the intentional convention in the comment block at the top of standings.ts ("byes count as wins for the receiver and that win does propagate into downstream opponents' Buchholz/Solkoff"). If the team prefers the strip-bye-wins convention, change computeRecord to track { wins, losses, bye_wins } and subtract bye_wins from the opponent's pool — but that's a behavior change, not a bug fix.

Test coverage notes

The integration-test suite under cloud/test/integration/tournament/ is one of the strongest aspects of the PR — every state-changing endpoint has a happy path + at least one rejection path, and the beta-gate.test.ts sweep tests the 404-not-403 contract across the whole tournament surface. Pure-function modules (pairing, standings, bracket, maps, canonical-map-options) each have direct unit tests with edge-case coverage (rematch-swap, bucket spillover, byes, cascade tie at cutoff, override_ranks validation).

Well-covered

  • Pairing edge cases — round 1, rematch-swap, bucket spillover, byes, exclusions (pairing.test.ts).
  • Standings — Buchholz, median-Buchholz (MB), forfeit/bye conventions, tie ranking (standings.test.ts).
  • Cascade tie at cutoff + override_ranks validation (transition-championship.test.ts).
  • Auto-advance scenarios, match_index population, multi-tournament /my-matches (flow.test.ts, generate-round.test.ts).
  • Rate limits — admin per-user, view per-IP — and audit-log structure (rate-limit-*.test.ts, audit-log.test.ts).
  • tournament_match_id upload binding + the LINKED_TO_ACTIVE_TOURNAMENT 409 from delete (upload-tournament-link.test.ts, delete-tournament-linked.test.ts).

Gaps worth filling

  • Discord-callback slot claim flow (cloud/src/auth.ts): the two-step UPDATE (pin by discord_id, fall back to discord_username) is the only path connecting Discord login → tournament participation. No tests exercise (a) first-time claim via username pins discord_id, (b) returning user re-claims via discord_id after a Discord rename, (c) substitution clears claim and the new user re-claims, (d) status='complete' tournaments don't auto-claim.
  • Bye-match upload attempt: no test asserts the worker's behavior when a participant POSTs a save with tournament_match_id referencing a match where status='bye'. The WRONG_HUMAN_COUNT path probably catches it, but the contract isn't pinned.
  • End-to-end substitution flow: admin edits discord_username on a claimed slot, the old user can no longer claim, the new user logs in and claims. The pieces are tested in isolation; the composition isn't.
  • Parallel last-match reports: maybeAdvanceAfterMatchReport's behavior when two reports complete the last round of a phase simultaneously. The concurrent-report safety on a single match (WHERE status='pending') is documented but the round/phase-level transition isn't.
  • parseAllowedMaps corruption surfacing MAP_CONFIG_INVALID at handleStartSwiss — pin the typed-error contract.

Strengths worth preserving

  • Pure-function algorithms isolated from D1. pairing, standings, bracket, maps, and rng take narrow SlotRef/MatchRef/config inputs and return plain values. Handlers map D1 rows to refs at the boundary. This is what makes the unit tests possible and the integration tests fast.
  • Deterministic mulberry32 RNG seeded on (tournament_id, phase, round). Same input → same pairings and map assignments. "Regenerate round" produces the same output until something explicit changes (e.g., new admin override). Integration tests rely on this.
  • Two-layer destructive-op guards. The games BEFORE DELETE trigger (migration 0013) + the LINKED_TO_ACTIVE_TOURNAMENT 409 in handleGameDelete is exactly the right pattern: a human-readable error from the worker, plus a DB-level safety net that protects future code paths that bypass the handler.
  • Migration discipline. Triggers over create-new-table for constraint changes (0009, 0013) avoid the PRAGMA foreign_keys toggling dance against tournament_matches' four FK references. Each migration has a heading comment that explains the change in context.
  • Centralized auth wrappers + 404-not-403 for feature hiding. authedTournament, requireTournamentBeta, requireTournamentAdmin applied in a consistent order — anon → beta → admin — so the beta-allowlist gate hides feature existence from outside the beta cohort.
  • Stable bracket slots survive substitution. tournament_slots persists across player swaps; standings are computed by querying tournament_matches against slot_id. No derived-state drift on retro-edit.
  • Match-modal shallow routing. +page.svelte:86-91 backs the modal off both page.state.match (push-state) and ?match= URL fallback, handles deep links and post-upload returns, and the cascade-deleted-ID effect at lines 118-120 closes the modal cleanly. Small surface, lots of UX value.
  • Two-phase swiss_seed rewrite for reorder. handleReorderSlots negates seeds then renumbers — an elegant sidestep of the UNIQUE (tournament_id, phase, division, swiss_seed) constraint without dropping the constraint.
  • Typed map-config errors. parseAllowedMaps surfaces JSON corruption as a typed MapConfigError with a specific code instead of a generic 500. Future-you debugging this is grateful.

Two of the candidate findings ("override_ranks duplicate slot refs", "handlePatchMatchMap doesn't intersect against allowed_map_scripts") overlap with that review's "functional bugs noted in passing" section and are not re-listed here.
  • cloud/package-lock.json dependency bump. Not inspected line-by-line; the preflight npm audit in the deploy CLI covers vulnerable versions.
  • Manual end-to-end against live Discord OAuth. The local Wrangler dev environment exercises the cookie + session path; the live Discord-app round-trip wasn't run as part of this review.
  • Visual / design QA. The SwissFlowBracket and ChampionshipBracketTree visualizations look right in the screenshots embedded in the implementation notes, but the review didn't render them against a real tournament.
