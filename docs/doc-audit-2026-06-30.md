This audit classifies every prose/spec doc under `docs/` into a three-state model and records the cleanup actions. It was produced by a two-phase process — a triage fan-out (one agent per cluster assigning verdicts) followed by an independent validation pass that re-checked every load-bearing claim against actual code. The validation pass is the load-bearing part: it caught raw greps that *looked* like contradictions but weren't, and three confidently-stated agent sub-claims that didn't survive contact with the code. The repeatable form of this process is the `doc-audit` skill (`.claude/skills/doc-audit/`).

## The three-state model

Every doc should be in exactly one of these states — there is no governed "just sitting in `docs/`" middle:

- **Authoritative** — linked from `CLAUDE.md`, claims verified against code; may be used as current guidance.
- **Historical** — carries a banner *and* lives in `docs/archive/`; a record, never read as instruction.
- **Deleted** — adds nothing a git archaeologist couldn't reconstruct.

The five verdicts below map onto that model: `keep`/`caveat` → authoritative; `archive` → historical; `extract` → lift the nugget then archive/delete; `delete` → gone.

A raw grep for `tournament_registrations` in `0006` initially *looked* like it contradicted the "table never built" claim — but the hit was the schema's own comment (`0006:8 -- * No tournament_registrations table — slots ARE the enrollment`). The agent was right; context was needed.

## Cross-cutting findings

1. **The authoritative allowlist is not self-verifying.** `tournament-feature-spec.md` rotted on `CLAUDE.md`'s Key-docs list. Re-verify the blessed set every audit.
2. **Format knowledge is durable; implementation framing is not.** The recurring caveat shape is a valuable save-file-format catalog wrapped in dead DuckDB/Rust code. Fix = banner the implementation half, keep the catalog — never delete.
3. **Security reviews on a public repo are a disclosure risk, not just stale docs.** Several security/code reviews mixed still-open findings with since-fixed ones — and the repo is public, so an in-tree review is a vuln roadmap. Resolution: extract open findings to **private** security advisories, then remove the review docs from HEAD (not merely archive them).
4. **Generated artifacts live in `docs/` undistinguished from prose.** `ux-review/{README,index}.html` are regenerable; `review.html` beside them is hand-written. A marker (or moving generated output out of `docs/`) stops future audits chasing "stale" build output.
5. **No provenance metadata.** Nothing carries a machine-readable `status` / `last-verified`. Frontmatter would make the next pass a diff, not a 30k-line re-read.
6. **The banner convention already works.** The ~5 docs that self-bannered were the easiest to adjudicate and least dangerous. Applying it consistently is the cheap, high-leverage move.

## Open findings → private advisories

The audit confirmed several still-open security findings buried in the (public) security/code reviews. Because the repo is public, the specifics are deliberately **not** written into this committed report — they were extracted to **private GitHub security advisories** (repo → Security → Advisories), covering the auth/CORS, session-lifecycle, response-header-hardening, and request-validation-ordering items. The one purely-cosmetic item (a modal focus-trap a11y gap) is the only candidate for a public issue.

The four source review docs — `security-review.html`, `asvs-l2-review.html`, `security-review-tournament.html`, `code-review-tournament.html` — were removed from HEAD (the repo is public; archiving them in-tree still discloses).
