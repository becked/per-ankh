docs/ Audit — Issue #39 Review

A file-by-file review of docs/ measured against the question: even though the Tauri/Rust/DuckDB stack is gone, does this doc still encode save-file format knowledge, XML quirks, parsing algorithms, or edge cases worth preserving for the TypeScript parser at src/lib/parser/?

Trigger: GitHub Issue #39 proposed deleting ~80 files from docs/. Jeff's instinct was that the issue deletes too aggressively. This audit confirms that instinct and proposes a more conservative cut.

1. Summary

108 files reviewed across docs/ and subdirectories. Each was classified KEEP, KEEP-with-caveats, EXTRACT, or DELETE against the save-file-knowledge rubric.

22 Keep Active runbooks, format reference, design docs
5 Keep w/ caveats Mostly accurate, parts stale — flag inline
43 Extract Has format/parsing nuggets worth saving
38 Delete Pure Tauri/Rust/DuckDB infra; no salvage

2. Headline finding

Issue #39 is right about most of the active docs, but wrong about ~40 of the proposed deletions. About half of the docs slated for deletion contain genuine save-file structure knowledge that won't exist anywhere else once they're gone: sentinel-value conventions, XML element hierarchies, version-evolution quirks, parsing-order constraints, real-world data patterns. The cost of deleting 80 files is bigger than the cost of keeping 60 with caveats — git history isn't a substitute when the LLM, the new contributor, and the format itself all benefit from having this stuff indexed and searchable.
Also discovered during the audit: Issue #39 didn't mention section-3.4-content-hashing.html, but it's actively cross-referenced from src/lib/SpriteMap.svelte — deleting it would break a live citation. And reference/parsing-and-importing.md has a correctness bug in its is_human rule (line ~540) that contradicts the actual TS parser; if we keep the doc, the rule needs to be fixed.

3. Methodology

Ten parallel exploration agents each read a batch of files end-to-end and returned, per file: a paragraph summary, a verdict (KEEP / EXTRACT / DELETE), and a list of any specific transferable nuggets. The rubric was identical across batches:

  • KEEP — still directly useful for the current SvelteKit/TS web app.
  • EXTRACT — contains save-file format insights, XML structure docs, parsing algorithm ideas, or edge cases that are language-agnostic and would help the current TS parser. Even one useful paragraph counts.
  • DELETE — pure Rust/DuckDB/Tauri implementation detail with no transferable knowledge.

The TypeScript parser at src/lib/parser/parsers/ was used as ground truth where docs claimed save-format facts (notably for the reference/ batch).

5. Save-format knowledge to preserve

Distilled from the EXTRACT pile — the load-bearing save-format facts that, if Issue #39 ran as-is, would no longer be documented anywhere searchable. These are the actual reason to keep ~half of what #39 wants to delete.

Sentinel value: -1 Used for "no player / unowned / anarchy / tribal / non-allied" across character, tile, tribe, and city entities. Parsers that don't filter -1 crash on get_player(-1). debugging-get-player-negative-one.md
Sentinel value: 2147483647 AI players have AIControlledToTurn="2147483647"; the active human has 0. But non-active humans in multiplayer have AIControlledToTurn > 0 — must also check OnlineID is non-empty. The TS parser does this; the old reference doc doesn't. plans/fix-human-player-detection.md + src/lib/parser/parsers/players.ts
Yields are fixed-point ×10 All yield integers are stored ×10 of display value. Always divide by 10 at the display layer. Design rationale (precision, multiplayer desync) explained. reference/yields.md, prospector-xml-data-sources-for-charts.md
Time-series sparse turn-keyed format <YieldRateHistory><YIELD_SCIENCE><T36>42</T36><T70>58</T70></YIELD_SCIENCE></YieldRateHistory> — only turns with changes are recorded. Charts need fill-forward. xml-parser-milestone-4-6-updates.md, prospector-xml-data-sources-for-charts.md
Player-keyed elements: <P.0> format Per-player values in many places (family/religion opinions, luxury trade, unit family). Element name is literal P.{playerIndex}. xml-parser-milestone-4-6-updates.md, plans/unit-ingestion.md, plans/city-data-coverage.md
Tile ownership history <OwnerHistory><T36>0</T36><T70>1</T70></OwnerHistory> — turn-keyed with -1 = unowned. Current owner = value at max turn. tile-ownership-implementation-hybrid.md
Greek dynasties as separate nations Seleucid/Ptolemaic/Antigonid show up as NATION_SELEUCUS, NATION_PTOLEMY, NATION_ANTIGONUS — not NATION_GREECE + dynasty. Format-stable across 266 saves / 3+ years. Normalization decision is open. greek-dynasty-normalization-issue.md, implementation-plan-dynasty-normalization.md
Memory list format: 2024 → 2025 change 2024: 5 separate lists (MemoryPlayerList, MemoryFamilyList, …) with numeric IDs. 2025: single MemoryList with string names. Parser needs both branches. backward-compatibility-memory-parser.md
DLCs vs Mods Version attribute encodes mods only (Version: 1.0.70671+mod1+mod2+...). DLCs are in a separate <GameContent> element with self-closing <DLC_*/> children. plans/separate-dlc-and-mods-parsing.md
Culture level is a string enum CULTURE_WEAK, CULTURE_DEVELOPING, … — not an integer. Schema/parser must handle as VARCHAR / string. issues/cities-tab-data-bugs.md
Unit production has version-specific shape UnitProductionCount (singular, older) vs UnitProductionCounts (plural, newer). Fall back when the new form is missing. issues/cities-tab-data-bugs.md, plans/city-data-coverage.md
FamilyClass is not a Root child FamilyClass elements live nested, not at root. Use descendants(), not children(). Maps families to archetypes (FAMILYCLASS_CHAMPIONS, _HUNTERS, _PATRONS, _CLERICS, _ARTISANS). issues/familyclass-parsing-issue.md
Individual units live inside <Tile> <Unit> elements with ID, Type, Player, Tribe, XP, TurnsSinceLastMove, CreateTurn, Facing, OriginalPlayer, PlayerFamily, Promotions, PromotionsAvailable, etc. — nested under <Tile>, not at Root level. Verified across 2022 / 2024 / 2025 / 2026 saves (counts 223–419 per save). Also <LastSeenUnits> contains snapshot copies for fog of war. Aggregate <UnitsProduced> / <UnitProductionCounts> are also present — they're for production analytics, not a substitute for real units. plans/unit-ingestion.md (canonical) + src/lib/parser/parsers/units.ts; verified by unzip-and-count across 4 saves
Optional <ID> in RelationshipData Some relationship records (self-relationships, incomplete data) omit the target ID child. Parser must handle as optional. debugging-character-relationships-id.md
Duplicate character rows exist in saves Marriages, stats, traits, relationships can appear with duplicate rows in source XML. Dedup pass (last-wins) required before insert. implementation/appender-deduplication.md
Parsing-order constraints Character parent FK (parents may appear later than children); tile → city ownership (city_id lives in City elements, not Tile); birth_city; ownership-history. All require two-pass / pre-resolution. hybrid_parser_migration_plan_v2.md, db-optimization-implementation-guide.md, future-optimization-tile-ownership.md
Within-import uniqueness Entities appear exactly once per save file. No need for in-file dedup logic except for the character-row edge cases noted above. on-conflict-investigation.md
Cities in anarchy: Player="-1" Cities being captured / in revolt can have Player = -1. Schema should allow nullable owner for transitional states. debugging-get-player-negative-one.md
High-value unparsed XML Verified-present in real saves but not yet parsed: city yields, legitimacy history, character marriages, LogData events at /Root/Player/PermanentLogList/LogData, MemoryData at /Root/Player/MemoryList/MemoryData. schema-save-format-alignment-analysis.md, next-parser-tasks.md, xml-parser-gap-analysis.md
Typical save scale 5 players, 449 characters, ~5,476 tiles, 43 cities, ~2,700 tiles with city ownership. Files 5–50 MB. XML parsing is ~6% of import time; DB operations are 94%. Use as design constraints & budgets. benchmark-results-*, performance-diagnosis-2025-11-07.md, performance-results-2025-11-07.md
Match-scoped entity IDs In the old DuckDB schema each match has independent ID sequences from 1. Composite PKs are (entity_id, match_id). Cloud schema uses different ID semantics — but the pattern explains the historical schema. schema-fix-composite-primary-keys.md, reports/parser_comparison.md
Enum cleanup conventions MAPCLASS_CoastalRainBasin → "Coastal Rain Basin", MAPSIZE_SMALLEST → "Duel", NATION_/RELIGION_/LEVEL_ prefix stripping plus regex/title-case rules. The TS app has formatEnum() for this; parser.py documents the original cases. parser.py, CLAUDE.md (formatEnum)
Schema migration archaeology schema.sql changelog (v2.0→v2.13) documents what was learned incrementally: data1/data2/data3 turned out to be VARCHAR not INT, units table added in v2.11, etc. Each change point reflects a save-format surprise. schema.sql lines 1024–1037
